## 1. 長期ゴールの特定方法(既存資産の再利用方法)

### 一次情報源: `user_fact_items`の`category=goals`

指示書の指示通り、新規のデータ収集機構は作らず、既存資産を横断参照する設計にし
た。長期ゴールの一次情報源は`user_fact_items`の`category=goals`とした。B17の調
査で判明した通り、`goals`は`importance_score`が固定1.0(全カテゴリ中最高)のカ
テゴリであり、これを明示的な長期目標の情報源として扱うことは既存の設計と整合的
である。

### 補助情報源: `sigmaris_decision_log`・`sigmaris_topic_log`

「目標に対して実際に何が起きているか」の証拠として、A3/B4で本稼働している
`sigmaris_decision_log`(直近50件)と、B6で本稼働している`sigmaris_topic_log`
(直近20件の話題ラベル)を横断参照した。

### 話題遷移を「暗黙の目標」として扱うことは見送った(判断根拠)

指示書は「B6の話題遷移で頻出する話題を、暗黙の目標として考慮できないか検討す
る」ことを求めていたが、**採用しなかった**。理由: 頻出する話題が必ずしも「目
標」を意味するとは限らず(単に最近よく話している話題というだけの可能性があ
る)、これを目標として扱うのは指示書自身が警告する「過剰な推測」に該当するリス
クが高いと判断した。代わりに、話題ログは「明示的な目標(goals)に対して、実際
の会話がどれだけ触れているか/離れているか」を判定するための**証拠**としてのみ
使い、目標そのものの情報源としては使わない設計にした。これにより「明示的な目標
情報を優先する」という指示書の要求を満たしている。

## 2. 整合性判定ロジックの実装詳細

### 週次バッチとしての実装(応答経路への影響なし)

`goal_alignment.py`に`extract_goal_alignment_flags(user_id)`を実装し、B2の
`consolidate_episodic_memory()`・B14の`extract_preference_patterns()`と全く同じ
構造(週次・fire-and-forget・LLMによる複数証拠の検証)にした。

### 判定基準: 1件では断定しない(B14の`_MIN_SUPPORTING_DECISIONS`を踏襲)

- 総決定数が3件未満(`_MIN_DECISIONS_FOR_ANALYSIS=3`、B14と同じ値)の場合、そ
  もそも抽出を試みない(`insufficient_data`として明示的に報告)。
- 候補として抽出された乖離は、根拠となる決定・話題のid(LLMが送信データの中か
  ら引用したもの)が実在するかを検証した上で、**最低2件以上**(`_MIN_SUPPORTING_
  EVIDENCE=2`、B14と同じ値)の裏付けがない限り棄却する。LLMが捏造したid・重複
  idは検証時にフィルタされ、フィルタ後に2件を下回れば棄却される(B7・B14と同じ
  防御パターン)。

### 過剰検出の防止(要件2)

LLMへの指示文(`_ANALYZE_PROMPT`)に、「些細な逸脱・一時的な寄り道・単発の出来
事は絶対に検出しないこと」「根拠が薄い場合は無理に乖離を捏造せず空リストにして
よい」と明示した。加えて、上記の複数証拠件数チェックがコード側の機械的な防御と
して機能する(LLMの判断だけに依存しない、B14と同じ二重の防御)。

### 検出内容と伝え方の分離(判断根拠)

抽出プロンプトには、`flag_statement`を「観察された事実の中立的な記述」にとどめ
るよう指示し、persona.md 9章のトーン調整は**注入時**(3章)に別途行う設計にし
た。これはB14の`pattern_statement`(事実の記述)と、注入時の確信度階層による言
い回し指示、という既存の役割分担をそのまま踏襲したものである。

## 3. 伝達ロジックの実装詳細(頻度制御、persona.md準拠の文面例)

### テーブル設計: `sigmaris_goal_alignment_flags`

B14の`sigmaris_user_preference_patterns`と同じ構造(`evidence_refs`・
`evidence_count`・`first_detected_at`・`last_confirmed_at`が複数回の週次実行に
わたって蓄積される)に加え、本タスク固有の`last_surfaced_at`列を追加した(頻度
制御用)。

### 頻度制御: 14日間のクールダウン(B3の48時間より長く設定した根拠)

`get_active_goal_alignment_flags()`は、`last_surfaced_at`が`null`または14日以
上前のフラグのみを返す(`_SURFACE_COOLDOWN_DAYS=14`)。B3の`active_inquiry.py`
の欠落項目確認(48時間)より大幅に長く設定した理由: 目標との乖離という話題は、
欠落している個人情報を尋ねるのに比べて、繰り返されるとより「小言」に感じられや
すいセンシティブな話題だと判断したため。

### 応答経路上での書き込みを避ける設計(要件5との整合)

「実際にプロンプトに注入した」時点でクールダウンを開始させる必要があるが、これ
を応答経路上で同期的にDB書き込みすることは要件5(レイテンシに影響を与えない)
に反する。B15(個人別放棄閾値)で確立した「保留状態を応答生成中に登録し、その後
の既存のfire-and-forgetバックグラウンド処理で実際の書き込みを行う」というパター
ンをそのまま踏襲した:

1. `_build_goal_alignment_context()`(応答経路上、読み取りのみ)が、実際にフラ
   グをプロンプトに含めた時点で`mark_pending_surfaced(flag_id)`を呼ぶ(プロセス
   内メモリへの登録のみ、DB書き込みなし)。
2. `_cognitive_layer_bg()`(既存のfire-and-forgetバックグラウンドタスク)に新
   設した`flush_pending_surfaced_flags()`が、保留中のフラグIDに対して実際に
   `last_surfaced_at`を書き込む。

**スレッドID単位ではなくグローバルな保留セットにした点がB3・B15との違い**(判断
根拠): B3・B15の保留状態はスレッド単位の会話文脈(「このスレッドでの直前の質
問・ヘッジへの返答」)に紐づく必要があるが、本タスクのフラグは特定のスレッドに
紐づく概念ではなく、システム全体で1人のユーザーに対して1回提示されたかどうかだ
けが問題になる。本システムが単一テナント(実質的に常に1つの会話ストリームしか
存在しない)であることを踏まえ、スレッドIDに紐づけない、よりシンプルなグローバ
ルなセットで十分と判断した。

### 1ターンあたり最大1件(判断根拠)

`_build_goal_alignment_context()`は、複数の有効なフラグが渡されても常に先頭
(最も証拠件数の多いもの)の1件のみを使う。持続的な指摘の羅列は「小言」感を強め
るリスクが高いため、1ターンにつき1つの穏やかな気づきに留める設計にした。

### persona.md 9章準拠の文面例

```
[長期ゴールとの整合性について(参考情報)]
以下は、直近の決定・話題が長期的な目標から乖離している可能性がある、という観察
です。persona.md 9章(制止する時のルール)に厳密に従い、「ちょっと待ってくださ
い。一度整理しましょう」「その前に確認したいことがあります」のような、確認・提
案の形でのみ触れてよいものです。「却下」という言い方や、断定的に「目標からズレ
ています」と指摘することは絶対にしないでください。会話の流れに合わない場合は、
無理に触れる必要はありません。
- 直近の決定はUI改善に集中しており収益化に直接つながる決定は見られない
```

## 5. テスト結果

実モデルAPIでの検証はできないため、mock/単体テストで確認した(2ファイル、計16
ケース、これまでのB群と同じくスクラッチテストとして作成・実行、backend/tests
配下にはコミットしていない)。

### `extract_goal_alignment_flags()`(6ケース)
- 目標が0件の場合、LLMを呼ばずinsufficient_dataを返すこと
- 決定件数が最低件数未満の場合も同様にinsufficient_dataを返すこと
- 2件の検証済み根拠を持つ候補が正しく保存されること
- 根拠1件のみの候補が棄却されること(要件3)
- LLMが送っていない架空のidを含む候補が、フィルタ後に閾値未満になり棄却される
  こと
- LLM呼び出し失敗時に例外を握りつぶすこと

### `flush_pending_surfaced_flags()`関連(4ケース)
- 保留がない場合、DB呼び出しが一切行われないこと
- 複数の保留フラグが正しく全て永続化され、保留セットがクリアされること
- 1件の書き込み失敗が他の書き込みをブロックしないこと
- `None`が渡された場合は無視されること

### `_build_goal_alignment_context()`(5ケース)
- フラグがない場合`None`を返すこと
- フラグが含まれる場合、persona.md 9章準拠の文面(「一度整理しましょう」「確認
  したいことがあります」)が含まれ、かつ「却下」という言い方を禁止する指示が含
  まれること(要件4)
- フラグを含めた時点で`mark_pending_surfaced`が呼ばれること
- 複数のフラグが渡されても先頭の1件のみが使われ、2件目は保留登録もされないこと
- 空白のみの`flag_statement`は`None`を返し、保留登録も行われないこと

### `_cognitive_layer_bg()`の配線(1ケース)
- `flush_pending_surfaced_flags()`が正しく呼ばれ、`goal_alignment_flags`キャッ
  シュが無効化されること

### 既存機能への非破壊確認(要件7)
`backend/tests/`の既存8テストを含む、これまでの全B群スクラッチテストを合わせた
181ケースを再実行し、全て成功することを確認した。

            テスト・既存回帰テスト再実行)

## 6. 気づいた懸念点・B9(ナレッジグラフ)着手前に検討すべき事項

- **本タスクで確立した「単一テナントのためスレッドIDに紐づけないグローバルな保
  留状態」という設計は、B9(ナレッジグラフ)が仮にエンティティ間の関係性を横断
  的に扱う場合にも応用できる可能性がある**: B9がもし複数の記憶ソースを横断して
  何らかの集約状態を持つ必要が生じた場合、本タスクや先行するB2/B6/B14/B15が既に
  確立した「週次バッチで抽出→service-roleグローバルテーブルに蓄積→フロントエ
  ンドのTTLキャッシュ経由で注入」というパイプライン全体が、そのまま再利用可能な
  テンプレートになりうる。
- **`sigmaris_topic_log`を「証拠」として使う設計は、話題ラベルの粒度に依存す
  る**: B6自身の報告書が指摘していた「話題ラベルの表現揺れ」が、本タスクの整合
  性判定の精度にも間接的に影響しうる。B9がナレッジグラフとして話題・エンティテ
  ィをより構造化する場合、この揺れの問題が自然に緩和される可能性がある。
- **B9着手前に検討すべき事項**: 本タスクを含め、B2・B6・B14・B15はいずれも「独
  立したservice-role専用のグローバルテーブル+週次抽出+TTLキャッシュ経由の注
  入」という同型のパターンを繰り返し実装してきた。B9がナレッジグラフとして複数
  のエンティティ・関係性を統一的に扱う設計になる場合、これらの個別テーブル群
  (`sigmaris_topic_log`・`sigmaris_user_preference_patterns`・`sigmaris_goal_
  alignment_flags`等)を集約・統合するのか、独立したまま維持するのかという設計
  判断が必要になる。本タスクでは既存資産の再利用を優先し新規統合は行っていない
  が、B9のスコープを検討する際にこの累積を俯瞰することを推奨する。
- **`_MIN_SUPPORTING_EVIDENCE=2`・`_SURFACE_COOLDOWN_DAYS=14`はいずれも初回設定
  であり、実際の週次バッチの発火頻度・目標の乖離が実際にどの程度の頻度で検出さ
  れるかは運用データでの観測が必要**(B11・B15の報告書でも同様の懸念を記載済
  み)。
