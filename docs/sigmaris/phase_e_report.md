## 1. サーバー環境のリソース状況の確認結果

### 1.1 実際に確認できたこと・できなかったことの明示

**結論から先に述べる: 本セッションからは、実サーバー(Ubuntu Server、ホスト名`sigmaris`)へのシェルアクセスができなかった。** 以下、確認過程を記録する。

1. **ネットワーク到達性は確認できた。** このセッションが動作している開発マシン(Windows)にTailscaleクライアントがインストール・接続済みであり(`tailscale status`で確認)、対象サーバー(`100.124.54.0`、hostname `sigmaris`)への接続状態は`active`だった。
2. **SSH接続を試みたが、認証情報が無かった。**
   - LAN経由(`ssh sigmaris@192.168.0.11`): 接続タイムアウト(このセッションの実行環境が、サーバーと同一LAN上に無いため——開発マシン側は自宅LANに居るはずだが、このセッション自体はその環境の外で動いている可能性が高い)。
   - Tailscale経由(`ssh sigmaris@100.124.54.0`): **TCP接続自体は成立し、SSHデーモンから`Permission denied (publickey,password)`という応答があった。** これは「サーバーは起動していて、SSHサービスも稼働している」ことの間接的な確認にはなるが、認証を通過できていないため、シェルコマンドは一切実行できていない。
   - このセッションの`~/.ssh/`には秘密鍵が存在せず(`known_hosts`のみ)、`sigmaris`ホスト向けの認証情報は最初から用意されていなかった。
3. **本番公開エンドポイントへの、読み取り専用の疎通確認のみ実施した。** `curl https://api.sigmaris.jp/`(GETリクエスト、認証無し)を実行し、`HTTP 404`(応答時間0.36秒)を確認した。これは`docs/cloudflare-tunnel.md`が記録している「ルート`/`は未実装のため404が期待値」という記述と一致しており、**FastAPI・Cloudflare Tunnel双方が現時点で正常に稼働していることの、間接的な確認**になる。この操作は本番の設定・データに一切影響を与えない(未認証の読み取り専用リクエスト1回のみ)。

**したがって、`df -h`・`free -h`・`docker --version`・`ss -tulpn`等の実際の出力は、本タスクでは取得できていない。** 依頼書の指示通り、推測でこれらの数値を埋めることはせず、1.3節に運用者が確認すべき具体的なコマンドを列挙する。

## 2. サンドボックス環境の、現実的な選択肢の整理

### 2.1 選択肢A: 静的な検証のみ(実際にコードを動かさない)

**内容**: 変更後のコードを、`backend/tests/`(既存16件)+ 新規に書かれたテストで実行し、既存テストが通るかだけを確認する。サーバープロセスとしては一切起動しない。

**実現性**: **極めて高い。追加のインフラ整備が一切不要。** 本セッションの`.venv`(`backend/.venv`)で`pytest`を実行するのと全く同じ方法が、そのままサーバー上でも(あるいはCI環境でも)使える。D-1〜D-3を含む、これまでの全Phaseで既に実践してきた検証方法そのものである。

**リスク**: 最も低い。テストプロセス自体が本番から完全に隔離されている(独立したPythonプロセス、本番DBへの書き込みは行わない限り発生しない——ただし`backend/tests/`の一部・scratchテストの一部は`unittest.mock`でHTTPクライアントをモックしており、実DBへの接続自体を経由しない設計になっているものが大半)。

**限界**: `codebase_size_report.md` 7章が既に指摘している通り、`backend/tests/`は16件のまま実質的に増えておらず、**実際の動作(サーバーが起動し、実際にHTTPリクエストを受けられるか、実際にDBへ読み書きできるか)は一切検証できない。** D-3が生成する`verifiable_metrics`(RC指標・Grounding指標等)のような、実際の会話ターンを経て初めて測定できる指標も、この方式では検証不可能。

**実装コスト**: ほぼゼロ(既存の`pytest`実行フローをそのまま流用するだけ)。

### 2.2 選択肢B: 別ポートでの、一時的な起動

**内容**: 本番(8000番)とは別のポート(例: 8001番)で、変更後のコードを一時的に起動し、実際にHTTPリクエストを送って動作確認する。

**実現性**: **高い。新しい技術の導入が不要。** `uvicorn app.main:app --port 8001`のように、既存の起動コマンドのポート番号を変えるだけで、原理的には即座に実現できる。ただし、以下の検討が必要。

**本番DBを共有するか、分離するかの検討(依頼書が明示的に求めた論点)**:

`docs/sigmaris/phase_c_full_report.md` 2.2節が確立した「本番データとの分離」パターン(**専用のSupabase Authアカウントを1つ作成し、既存のRLS(`auth.uid() = user_id`)による分離だけで、新しい分離ロジックを一切実装せずに完全な分離を得る**)が、そのままここでも応用できると判断した。**判断根拠**: C-full-1が別のテーブル/スキーマ案を検討した上で見送った理由(「`search_relevant_memories()`等をそのテーブル向けに作り直す必要があり、シグマリスの実際のパイプラインを評価するという目的から外れてしまう」)は、Phase Eのサンドボックス検証にもそのまま当てはまる。**別スキーマ・別DBを新設するのではなく、既存のSupabaseプロジェクト内に、検証専用のSupabase Authアカウントを追加する方式を推奨する。**

ただし、Phase Eの検証対象がD-3のhandoffに含まれる`target_files`(未実装、コード変更そのもの)である以上、C-full-1(既存パイプラインへの読み取り専用に近い評価)より一段階リスクが高い——**検証用アカウントであっても、変更後のコードに未知のバグがあれば、そのアカウントの記憶データを破壊しうる。** この点はC-full-1のスコープには無かった、Phase E固有の追加リスクとして明記する。

**マイグレーション(DBスキーマ変更)を伴う仮説の扱い**: 別ポートでの起動は、アプリケーションコードの分離はできるが、**Supabase(PostgreSQL)自体は本番と共有**である以上、マイグレーションを含む仮説の検証には使えない(3章で詳述)。

**リスク**: 中程度。ポートが分離されていても、環境変数の設定ミス(例: 検証用の`.env`のつもりが本番の`SUPABASE_SERVICE_ROLE_KEY`を指してしまう)によって、本番DBへ意図せず影響を与えるリスクがある。C-full-1の「多層の安全機構」(ベンチマークJWTから解決したuser_idと本番のuser_idが一致しないかのクロスチェック)が、ここでも直接の前例として使える。

**実装コスト**: 低〜中。検証用Supabase Authアカウントの作成(運用者による1回の手動作業、C-full-1と同じ)、検証用の`.env`ファイルの用意、起動スクリプトの整備程度で済む。

### 2.3 選択肢C: Dockerコンテナでの、隔離実行

**内容**: 完全に隔離されたコンテナ内で、変更後のコードを試す。

**実現性**: **未確認(1.3節参照)。** リポジトリに`backend/Dockerfile`が既に存在するため、Dockerさえインストールされていれば`docker build`自体はすぐに試せる状態にはある。ただし、実際にサーバーにDockerがインストールされているかどうかは、この調査では確認できなかった。

**導入コストの評価**:
- Dockerが未インストールの場合、`apt install docker.io`(またはDocker公式リポジトリからの導入)自体は定型的な作業だが、**本番稼働中のサーバーへの新しいシステムパッケージの追加**であり、依頼書「本番の設定に影響を与える操作は一切行わないこと」という制約の範囲内で本タスク自体が実施することはできない——これは運用者の判断・作業になる。
- コンテナ内からSupabase(外部API)への接続は、ネットワーク的には問題なく可能(HTTPS経由のため、Docker特有の設定は基本的に不要)。
- 将来的にOllama/ローカルLLM(GTX 1660)をコンテナ内で使う場合は、NVIDIA Container Toolkitの追加導入が必要になる——ただし、**現時点で本番自体がOllamaを実運用していない**(`LOCAL_LLM_ENABLED`デフォルトfalse)ため、Phase Eの当面のスコープ(LLM API呼び出しはOpenAI経由が主)ではこの追加コストは必須ではない。

**得られる安全性**: 最も高い。プロセス空間・ファイルシステムが本番と完全に分離され、コンテナを削除するだけで痕跡なく破棄できる。選択肢Bで挙げた「環境変数の設定ミスで本番に影響する」リスクも、コンテナのネットワーク設定(本番DBへの到達性を意図的に絞る等)で構造的に低減できる。

**リスク**: 選択肢A・Bと比較すると、**新しい技術の導入自体がリスク**になりうる(未検証のDocker設定が、意図せずホストのリソースを圧迫する、ネットワーク設定のミスで想定外の通信が発生する等)。GTX 1660 6GBという限られたVRAMを、コンテナ経由で正しく制御できるかも、実際に試すまでは未知数。

**実装コスト**: 中〜高。Dockerのインストール確認・(必要なら)導入、`Dockerfile`の検証・本番相当の環境変数の安全な注入方法の設計、コンテナのライフサイクル管理(検証後の自動破棄等)の実装が必要。

## 3. 「ロールバック」を、どう実現するか

### 3.1 コード変更のロールバック(比較的容易)

D-3の`build_phase_e_handoff()`が設計した`target_files`(未実装のプレースホルダ)が実際に使われる段階になれば、コード変更自体は**gitのブランチ管理でロールバックできる**。このセッションが一貫して実践してきたワークフロー(作業ブランチで実装→テスト→mainへマージ)がそのまま使える。「ロールバック」は、単純にはそのマージコミットを`git revert`する、あるいはマージ前の状態を指すコミットへ本番のデプロイ対象を戻すだけで足りる。**この部分に、新しい仕組みは必要ない。**

### 3.2 DBスキーマ変更(マイグレーション)を含む仮説のロールバック(慎重な検討が必要、依頼書が明示的に警告した通り)

これがPhase Eの最も難しい論点だと判断した。理由は以下の通り。

1. **このコードベースの既存の運用方針そのものが、「マイグレーションは作成のみ、適用は運用者の手動判断」という設計になっている**(Phase R〜D、全タスクの「注意事項」で一貫)。つまり、現状「マイグレーションの自動ロールバック」に相当する仕組みは、Phase E以前のどのフェーズにも存在しない。
2. **PostgreSQLのDDL変更は、一般に無条件に安全に戻せるとは限らない。** 列の追加(`ALTER TABLE ... ADD COLUMN`)は比較的戻しやすい(`DROP COLUMN`)が、データを含む変更(例: 既存列の型変更、NOT NULL制約の追加でデフォルト値を持たない既存行がある場合)は、ロールバック用のSQLを自動生成すること自体が難しい。
3. **Phase A4のCAS(楽観ロック)パターンは、この問題を直接解決しない。** CASは「同時書き込みの競合」を検知する仕組みであり、「DDL変更を安全に戻す」こととは性質が異なる。ただし、CASが確立した**「マイグレーション適用前・適用後のどちらでもコードが安全に動く」という設計原則**(A4報告書3章: 「未適用の間はCAS機能が単に無効なだけで、既存機能への影響はゼロ」)は、Phase Eのマイグレーション設計にも直接応用できる重要な前例だと判断した——**新しい列は常にNULL許容・デフォルト値ありで追加し、旧コードが新しい列の存在を知らなくても動き続ける形にする**、という制約を、Phase Eが検証すべき「安全なマイグレーション」の必須条件として明文化する価値がある。

**推奨する扱い**(判断根拠、依頼書「慎重な検討が必要であることを報告に明記すること」への対応): マイグレーションを含む仮説は、**選択肢Bの「別ポート起動」だけでは検証しきれない**(Supabase自体が本番と共有のため、マイグレーションを適用した時点で本番にも影響する)。このため、**マイグレーションを含む仮説は、当面「常にrequires_special_review相当の特別扱いとし、自動的なサンドボックス検証の対象からは除外し、運用者による手動レビュー(SQLの内容を人間が読んで判断)を必須とする」という運用ルールを、Phase Eの設計に組み込むことを推奨する。** これは、D-2/D-3が既に確立した「Constitutionに触れる可能性のある仮説は別枠で慎重に扱う」という設計哲学(`sigmaris_hypotheses.requires_special_review`)を、マイグレーションという別の軸に横展開する形になる。

将来的に、Supabaseの別プロジェクト(完全に独立したPostgreSQLインスタンス)を検証用に用意できれば、マイグレーションの実適用まで含めた自動検証が可能になるが、これは無料枠の制約・追加の運用コストを伴うため、**本タスクでは推奨案に含めない**(4章で「将来の発展」として触れるにとどめる)。

## 4. 推奨案

### 4.1 段階的導入を推奨する(判断根拠)

依頼書が示唆した通り、**選択肢A→B→Cの順で段階的に導入することを推奨する。** 理由は以下の3点。

1. **選択肢Aは追加のインフラ整備が不要で、即座に導入できる。** Phase D-3までで既に実践してきたテスト手法の延長であり、Phase Eの最初の一歩として自然。
2. **選択肢Cの前提(Dockerの実際の可用性)が、本タスクでは確認できていない。** 未確認の前提に依存した設計を最終形として確定させるのは時期尚早であり、1.4節の確認結果を待ってから判断すべき。
3. **選択肢Bは、C-full-1が既に確立した「専用Authアカウント+既存RLS」パターンをそのまま転用でき、新しい分離ロジックの実装コストが低い。** 選択肢Cへ発展させる場合も、選択肢Bで確立した「検証用Authアカウント」の仕組みはそのまま使い続けられる(コンテナの中でどのDBアカウントを使うか、という話に置き換わるだけ)。

### 4.2 推奨する段階構成

## 8. 静的検証の実装詳細

新設: `backend/app/services/static_verification.py`(純粋関数)・`static_verification_runner.py`(I/O)・`static_verification_store.py`(永続化)。既存の3層分離パターンを踏襲した。

### 8.1 選んだ検証方式(判断根拠、依頼書「リスクの低い方法から選ぶ」への対応)

依頼書が例示した2つの方式(a. 既存テストをそのまま実行するベースライン確認、b. より踏み込んだシミュレーション)のうち、**両方を組み合わせたが、bは「仮の変更を一時的なブランチに適用してテストを走らせる」という、依頼書が慎重な設計を求めた方式ではなく、「テキストレベルの静的な突き合わせ」という、より保守的な形にとどめた。**

1. **ベースライン確認**: `backend/tests/`を、一切変更せずそのまま`subprocess`で`pytest`実行する(`run_baseline_test_suite()`)。既存のテストファイル・アプリケーションコードのいずれも書き換えない、読み取り専用の確認。
2. **カバレッジ照合(静的解析、実行なし)**: 仮説の自由文(`title`/`what_is_problem`/`why_problem`/`how_to_improve`/`source_evidence.title`)から、Sigmarisのモジュール名らしきトークンを抽出し(`extract_candidate_modules()`)、`backend/tests/`配下の各テストファイルの`import`文を`ast.parse()`で静的解析して得た「どのモジュールがどのテストでカバーされているか」の索引(`build_test_coverage_index()`)と突き合わせる。

**判断根拠(「仮の変更を一時ブランチに適用してテストを走らせる」方式を選ばなかった理由)**: 依頼書は「具体的な実装方法は、慎重に設計すること」と述べているが、以下の理由から、この方式は本タスク(E-1)の時点では時期尚早と判断した。
- D-2の生成プロンプトは、仮説から実際のコード差分(diff)を出力させていない——`how_to_improve`は「実装の詳細設計ではなく、提案の段階」(D-2報告書、11.1節)にとどまる自然文であり、これを自動的に「一時的なコード変更」へ変換する処理自体が、依頼書が禁じる「実際にコードを書き換える仕組み」に該当してしまう。
- 依頼書自身が「これは、次の段階(E-2以降)で、より慎重に設計する」と明記しており、E-1の時点でこれを実装することは、依頼書が意図する段階分けに反すると判断した。

### 8.2 「合格/不合格」の二値判定にしなかった判断根拠(重要)

依頼書は検証結果の例として「合格、不合格、検証不能等」を挙げているが、**実際にコードを一切実行しない静的検証だけでは、ある仮説が「正しい」ことも「間違っている」ことも証明できない。** そのため、この手法で誠実に主張できる範囲にとどめた、以下の3値(+除外1値)の判定にした。

| verdict | 意味 |
|---|---|
| `excluded_migration` | マイグレーションに言及しているため、本パイプラインの対象外(9.2節) |
| `baseline_unhealthy` | `backend/tests/`自体が現在失敗している。個別の仮説を評価しても無意味なため、この一値に統一する |
| `insufficient_signal` | ベースラインは健全だが、対象領域と推定されるモジュールに既存テストが見つからない——**依頼書の「合格」に相当する最も楽観的な結果でも、あくまで参考情報**であることを、名前自体に反映した |
| `baseline_healthy_with_coverage` | ベースラインが健全で、対象領域に既存テストが1件以上存在する |

判断根拠: Phase G-5のSearch Trigger Rateが自らを「下限近似値」と明記し、Phase R-2のRC-1が`evaluated_not_promoted`を安易にhealthy/unhealthyと決めつけなかったのと同じ、「この手法で言えることの限界を、判定結果自体に正直に反映する」という、このコードベース一貫の設計哲学を踏襲した。

## 9. 検証対象の絞り込みロジック

### 9.1 D-3のnormalトラックからの取得(要件1)

`hypothesis_prioritization_store.get_recent_prioritization_run(limit=200)`をそのまま呼び出し、**直近1回分の`run_at`にのみ絞り込んだ上で**、`track == "normal"`かつ`phase_e_handoff`が存在する行のみを、`priority_rank`昇順(既にDB側のクエリで保証済み)で上位`limit`件取得する(`_latest_run_normal_track()`)。

**`track == "normal"`の絞り込みを、D-3の構造的な分離に加えて、E-1側でも明示的に再確認した判断根拠**: D-3の`prioritize_hypotheses()`は、既に`requires_special_review`の仮説を`special_review_track`へ構造的に分離しており(`phase_e_handoff`も`None`固定)、原理的にはE-1が読み取るデータには混入し得ない。それでも依頼書が「この静的検証パイプラインの対象からも、明確に除外すること」と重ねて強調している以上、**単一の防御層に頼らず、E-1自身でも`track`フィールドを直接確認するガードを設けた**——多層防御は、C-full-1がベンチマーク専用アカウントの分離で採用した設計判断(「本番側の認証情報と一致しないかのクロスチェック」)と同じ考え方である。

### 9.2 マイグレーションを伴う仮説の除外(要件2)

`mentions_migration()`——D-2の`rule_based_safety_flag()`と同じ「キーワード一致、安全側に倒す」方式。仮説の`title`/`what_is_problem`/`why_problem`/`how_to_improve`に、`マイグレーション`/`migration`/`ALTER TABLE`/`CREATE TABLE`/`スキーマ変更`/`supabase/migrations`/`.sql`等のキーワードが1件でも含まれていれば、`excluded_migration`として即座に除外し、ベースライン確認・カバレッジ照合のいずれも行わない。

**判断根拠**: `docs/sigmaris/phase_e_report.md` 3.2節が既に到達した結論(「マイグレーションを含む仮説は、当面`requires_special_review`相当の特別扱いとし、自動的なサンドボックス検証の対象からは除外し、運用者による手動レビューを必須とする」)を、そのままE-1の実装へ落とし込んだ——新しい判断基準を作らず、直前の調査タスク自身が下した結論を再利用した。

`sigmaris_static_verifications`(新設、未適用)。D-2(`sigmaris_hypotheses`)・D-3(`sigmaris_hypothesis_priorities`)と同じ「追記専用ログ、仮説1件=1行」の設計を踏襲した。

実行方法: `python scripts/run_static_verification.py [--limit N] [--dry-run]`。

## 12. 気づいた懸念点・次のステップ(E-2: 別ポートでの動的検証)に向けた申し送り事項

1. **カバレッジ照合は、非常に薄いシグナルにとどまる(7章で先述)。** `backend/tests/`が5モジュールしかカバーしていない現状では、D-1〜D-3を含むこのセッションで実装されたほとんどのモジュールを対象にした仮説が、`insufficient_signal`(参考情報なし)になることが予想される。**これは静的検証パイプラインの実装上の不備ではなく、`backend/tests/`自体の薄さという、Phase E以前から存在する構造的な制約の表れである**(`codebase_size_report.md` 7章2項の懸念点がそのまま顕在化した形)。この状況を改善するには、E-1自体の拡張ではなく、各Phaseのscratchテストの一部を`backend/tests/`へ昇格させる運用ルールの検討が必要になると考えられる——これは本タスクのスコープ外だが、次タスク以降で検討する価値がある。
2. **`extract_candidate_modules()`は、D-2が対象ファイル名を出力していないことの代償として、仮説の自由文からモジュール名を推測する、本質的に不正確なヒューリスティックである。** 誤検出(無関係な語がたまたまモジュール名と一致する)・見逃し(モジュール名が言い換えられていて一致しない)の両方が起こりうる。D-2の生成プロンプトに、対象と推定されるファイル名を明示的に出力させる項目を追加すれば(D-3の`target_files`プレースホルダを、D-2の時点で部分的に埋める形)、この推定の精度を大きく改善できる可能性がある——ただし、これはD-2への変更を伴うため、本タスクの範囲外とし、次タスクへの提案にとどめる。
3. **`insufficient_signal`が多数を占める場合、優先順位付け(D-3)との組み合わせで、どの仮説から着手すべきかの実質的な判断材料が乏しくなる。** E-2(別ポートでの動的検証)が実装されれば、実際にAPIを叩いて動作確認できるようになり、この限界の多くは解消される見込みだが、E-1単体では「実装前に、この領域には既存の回帰テストがあるか」という補助的な情報提供にとどまることを、運用者が理解した上で使う必要がある。
4. **ベースライン確認(`pytest`のsubprocess実行)には、タイムアウト(デフォルト120秒)を設けた。** `backend/tests/`が将来大幅に増えた場合、この値の見直しが必要になる可能性がある——現状(16件、実測1.5秒程度)では十分な余裕がある。
5. **E-2(別ポートでの動的検証)に向けて**: `docs/sigmaris/phase_e_report.md` 4.2節・5章が既に設計した通り、E-2はC-full-1の「専用Supabase Authアカウント+既存RLS」パターンを転用し、実際にAPIを起動して動作確認する段階になる。E-1が今回構築した`sigmaris_static_verifications`のverdict(特に`baseline_healthy_with_coverage`の仮説)を、E-2が「動的検証を優先的に試す候補」の絞り込みに使うという接続方法が、次タスクの自然な入り口になると考えられる。
