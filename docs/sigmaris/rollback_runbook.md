# ロールバック運用手順書(Phase E-4)

**作成日**: 2026-07-30
**目的**: Phase Eのサンドボックス検証(E-1・E-2)で問題が見つかった場合、または本番適用後に問題が判明した場合に、**確実に元の状態へ戻すための、具体的な手順**をまとめる。**本タスク(E-4)は、新しい自動ロールバックの仕組みを実装しない。** ここに書かれているのは、既存のgit・Supabaseの仕組みを使った、人間が手動で実行する手順である。

## 1. コードレベルのロールバック(依頼書2章)

### 1.1 前提: このリポジトリの既存ワークフローは、既にロールバック可能な形になっている

このセッションのPhase R以降、全てのタスクは一貫して以下の手順で進められてきた。

1. `main`から作業用ブランチを新規作成する
2. 実装・テストをそのブランチ上で行う
3. `git merge --no-ff <ブランチ名>`で`main`へマージする(**常に`--no-ff`——マージコミットが必ず1つ作られる**)
4. `git push origin main`

この「常に`--no-ff`でマージする」という既存の慣習自体が、**ロールバックを容易にする設計になっている**——マージコミットが必ず存在するため、`git revert -m 1 <マージコミットのハッシュ>`一発で、そのタスク一式をまとめて打ち消せる。新しい仕組みは不要で、**既存の運用を続けるだけで、ロールバック可能性は既に確保されている**、というのが本タスクの結論である。

### 1.2 具体的なロールバック手順

#### ケースA: 直近のマージを、まだ誰も手元にpullしていない・本番へデプロイしていない場合

```bash
# 直近のマージコミットを確認
git log --oneline -5

# そのマージコミット自体を打ち消す新しいコミットを作る(推奨・安全)
git revert -m 1 <マージコミットのハッシュ>
git push origin main
```

`-m 1`は「マージの1番目の親(=マージ前のmain)を正とする」という指定——`--no-ff`マージ特有の、必ず指定が必要なオプション。

#### ケースB: 特定のPhaseまるごとではなく、その中の1コミットだけを戻したい場合

```bash
git log --oneline <対象ブランチのマージ範囲>
git revert <個別のコミットハッシュ>
git push origin main
```

#### ケースC(非推奨、破壊的): 履歴そのものを巻き戻したい場合

```bash
# 注意: 既にpushされたmainの履歴を書き換えることになる。
# 他の開発者・自動デプロイが既にそのコミットを前提にしている場合は使わないこと。
git reset --hard <戻したいコミットのハッシュ>
git push --force origin main
```

**このケースCは、本番稼働中のシステムに対しては、原則として使わないことを推奨する。** ケースA(`git revert`)は履歴を保持したまま前の状態に戻すため、「いつ・何が・なぜ戻されたか」がそのまま記録に残る——これは、このセッション全体が一貫して重視してきた「判断根拠を残す」という方針とも合致する。

### 1.3 本番サーバーへの反映

`docs/sigmaris/phase_jwt_persistence_report.md` 4章が既に記録している、本番`sigmaris-backend`サービスへの適用手順(通常のデプロイ手順)がそのまま使える。

```bash
# サーバー側(Tailscale SSH経由)
ssh sigmaris@100.124.54.0
cd <リポジトリのパス>
git pull origin main
sudo systemctl restart sigmaris-backend
```

ロールバック時も同じ手順(`git pull`で、revertされた後のmainを取得→再起動)で反映できる——新しい再起動手順を用意する必要はない。

## 2. マイグレーションのロールバック方針(依頼書3章)

### 2.1 調査結果: 既存56件のマイグレーションは、いずれも"戻し方"を持たない

`supabase/migrations/`配下の全56ファイルを確認した(`grep -ril "rollback\|down migration\|-- down\|revert"`)。**"down"migration・ロールバック手順のコメントを持つファイルは、1件も存在しなかった。** 別ディレクトリ(`supabase/migrations/down/`等)や、別命名規則(`*_down.sql`)も存在しない。**このコードベースのマイグレーションは、設計当初から一貫して「一方向(up only)」であり、これはPhase A以前から続く既存の設計方針である。**

### 2.2 リスク別の分類(実施した調査、判断根拠つき)

各マイグレーションが実際に行っているDDL/DML操作を機械的に分類した(`grep -oiE`による集計)。

| カテゴリ | 該当件数 | 内容 | 手動での戻しやすさ |
|---|---:|---|---|
| **A. 純粋な追加のみ** | 約49件 | `CREATE TABLE`・`ALTER TABLE ADD COLUMN`・`CREATE INDEX`・`CREATE POLICY`・`CREATE TRIGGER`のみ | **易しい。** `DROP TABLE`/`DROP COLUMN`/`DROP INDEX`等、対になる操作が機械的に導出できる。ただし列を削除する場合、その列に既に入っているデータは失われる(後述2.3節) |
| **B. 関数の再定義(DROP FUNCTION→CREATE FUNCTION)** | 4件(`202607090031`・`202607110033`・`202607150037`・`202607220045`) | Postgresの制約(`CREATE OR REPLACE`は`RETURNS TABLE`への列追加を許さない)により、`search_fact_memory`/`search_fact_memory_trgm`という2つのRPCを、B群の機能追加のたびに`DROP`+`CREATE`し直している | **易しい。** データそのものは一切変更しない、関数定義の差し替えのみ。**1つ前のマイグレーションファイル(git履歴上)に書かれている関数定義を、そのまま再実行すれば戻せる**——実質的に、git履歴自体が"down migration"の役割を果たしている |
| **C. 既存データの書き換え(UPDATE/backfill)** | 2件(`202604230009_align_surface_travel_modes.sql`・`202606270019_trend_memory.sql`) | 既存行の値を新しい正規化された値へ一括変更(例: `travel_mode`の`'driving'`→`'car'`統合、`importance_score`/`privacy_level`のバックフィル) | **【重要、懸念点として明記】難しい。** 上書き前の元の値が、この2件のマイグレーション自体には残っていない。ロールバックするには、適用前のデータベースの状態(バックアップ)が必要——マイグレーションファイルを読むだけでは戻せない |
| **D. 列・テーブルの削除** | 0件 | `DROP COLUMN`・`DROP TABLE`は、調査した56件のいずれにも存在しない | (該当なし) |

**結論**: カテゴリDが0件であることは良い兆候(このコードベースは、これまで一度も「後戻りできない形でのデータ喪失」を伴うマイグレーションを行っていない)。一方、**カテゴリC(2件)は、既存のマイグレーションファイルだけを見ても元に戻せない、真のロールバック不能地帯である**ことを、懸念事項として明記する(依頼書3章「今後の懸念事項として、明確に報告すること」への対応)。

### 2.3 カテゴリAでも注意が必要なケース(列削除)

カテゴリA(純粋な追加のみ)は「戻しやすい」と分類したが、これは「戻す操作自体は機械的に書ける」という意味であり、**列を追加してから何らかのデータがその列に書き込まれた後にロールバック(列削除)すると、そのデータは失われる**。例えば`supabase/migrations/202607260053_hypotheses.sql`(`sigmaris_hypotheses`)適用後、実際に仮説が生成・保存された後にこのマイグレーション自体をロールバックすると、生成済みの仮説データは全て失われる。**「戻す操作が書けること」と「戻しても実害が無いこと」は別**、という点を、今後の運用で混同しないよう明記しておく。

### 2.4 今後の新しいマイグレーションに推奨する方針(依頼書「実装は次の機会でよい」)

**本タスクでは実装しない、将来への推奨事項として記載する。**

1. **新しいマイグレーションファイルには、可能な限り、対応する"戻し方"をコメントとして併記することを推奨する。** 例:
   ```sql
   -- Rollback: drop table if exists public.sigmaris_xxx cascade;
   ```
   これは実行可能なSQLとして別ファイルに分離する必要はなく、**同じファイル内にコメントとして残すだけでも、次に見た人(将来のセッション、または運用者自身)が判断する材料になる**——このコードベース全体が一貫して実践してきた「判断根拠をコメントに残す」という文化の延長。
2. **既存データを書き換える(カテゴリC相当の)マイグレーションを新たに書く場合は、書き換え前の値を、コメントとして明示的に記録するか、書き換え対象の行を一時テーブルへ退避するような一手間を検討することを推奨する。** 本タスクの調査で分かった通り、この種のマイグレーションが最もロールバックが難しいカテゴリであるため。
3. **本タスクでは、既存56件のマイグレーションファイルへ、遡って"戻し方"コメントを追記することはしない**(依頼書「既存の全マイグレーションを遡って直すことは、本タスクでは範囲外とする」に従った)。

## 3. Phase D-4以降が新設するマイグレーションへの、実際の適用

`supabase/migrations/202607300057_migration_review_queue.sql`(本タスクで新設)を含め、Phase D〜Eを通じて作成された全てのマイグレーションは、**作成のみで、適用は運用者の手動判断に委ねられている**(このセッション全体で一貫した方針)。運用者は、Supabase Dashboardまたは`psql`で、`supabase/migrations/`配下の未適用ファイルを、ファイル名の日付順に適用することを推奨する。

適用状況の確認方法は、`docs/sigmaris/bug_inventory.md` 8章が既に確立した検証クエリ(過去に44件全件の適用状況を確認した実績がある)がそのまま使える。
