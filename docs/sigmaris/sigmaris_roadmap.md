# シグマリス計画 統合ロードマップ

**作成日**: 2026-07-03
**前提**: 2026-07-02実施の監査レポート(`docs/sigmaris/global_state_migration_audit.md`)を踏まえた再設計。
**方針転換のポイント**:
- Mem0は導入しない。既存の`user_fact_items`/`sigmaris_decision_log`/`sigmaris_experience`/pgvector RAGを本稼働させる自前拡張路線
- ベンチマークは自作せず、LongMemEval/LoCoMo(公開データセット)をそのまま流用して客観性を担保
- 「業界的に個人でここまでやったか」の訴求は、ベンチマーク絶対値の最大化よりも、クローズドループの完全動作+再現可能な公開+リソース制約下の効率性で狙う
- パーソナルAIとしての体感性能(海星さん個人への適応精度)とベンチマーク数値の両方を並行して改善する

---

## 全体構成(v2: 小さなPDCA方式に変更)

```
Phase A: 基盤修正(旧Step0〜5)
   ↓
Phase C-mini: 最小限の評価基盤のみ先に作る(memory_f1 / rag_ndcg / response_error_rate)
   ↓
┌─────────────────────────────┐
│  B機能を1つ実装 → C-miniで評価 → 次のB   │ ← このループをB群全体で反復
└─────────────────────────────┘
   ↓
Phase C-full: 測定基盤を完成させる(独自指標全部 + 公開ベンチマーク運用)
   ↓
Phase D〜H: 自己改良システム本体(旧タスク3 Phase2〜6)
```

**v1からの変更点**: 当初「Phase Cを丸ごと完成させてからPhase Bへ」という設計だったが、これだと評価基盤の構築自体が長期化し、Bへの着手が遅れる。ChatGPTのレビューを踏まえ、「A完了→最小限の評価基盤→B1つ実装→評価→次のB」という小さなPDCAループに変更。B機能を1つ実装するたびに測定し、効果のないものは早期に見切りをつけられるようにする。

**依存関係の鉄則**: Phase C-fullが完成する前にPhase D以降(自己改良ループ)には進まない。「良くなったか測れない」状態で自己改良を回すと、改良の効果を検証できない。ただしB群の実装中はC-miniの3指標だけで都度評価し、B群完了後にC-fullで公開ベンチマーク込みの本格評価を行う。

## Phase A: 基盤修正

**目的**: スレッド依存の解消・応答速度の改善・記憶の土台を正常化する。ここが終わらないと以降の高度な機能が乗らない。

| # | タスク | 内容 | 優先度 |
|---|---|---|---|
| A0 | 書き込み経路の一本化 | `frontend/src/lib/chat-threads.ts`(Next.js直接Supabase書き込み)と`backend/app/services/app_chat_data.py`(FastAPI経由)の二重書き込み経路を統合。どちらか一方を正の経路にする | 最優先(土台) |
| A1 | セッション継続方式の実装 | `previous_response_id`の活用、または「直近ターンのみ生送信+それ以前はグローバル状態(decision_log/active_topics相当)で代替」する方式を実装。現状`input`配列を毎ターン全部送信しており線形に増大している問題を解消 | 最優先(体感インパクト最大) |
| A2 | プロンプト構造の並べ替え | `chat_prompts.py::build_system_prompt()`の結合順序を「固定・長大な指示(rules)を先頭、可変要素(現在時刻・意図分類結果・ユーザー文脈)を末尾」に変更。OpenAIのオートキャッシュ(プレフィックス一致)を有効化する | 高 |
| A3 | `sigmaris_decision_log`の本稼働 | 現状`chat_turn:xxxx`という空疎な記録のみの状態から、実際の決定内容(何を決めたか・方針変更の有無)を毎ターン後に非同期で問い合わせて記録する形に作り直す | 高 |
| A4 | 同時書き込みの排他制御 | `replace_chat_messages()`(全削除→全INSERT方式・楽観ロックなし)に、`updated_at`/バージョン列によるCAS、または行レベルロックを導入 | 中 |
| A5 | RAGの`LOCAL_LLM_ENABLED`依存見直し | `LOCAL_LLM_ENABLED=false`時にpgvector検索が丸ごとスキップされる現状の分岐を見直し、OpenAI利用時も記憶検索が機能するようにする | 中 |

## Phase B: Mem0を超えるための記憶拡張機能

**目的**: Mem0(汎用記憶ライブラリ)にない要素で質的に上回る。公開指標側とパーソナル性能側の両方に効くロジックを含む。

### B系(基本の6機能)

| # | 機能 | 概要 | 工数目安 |
|---|---|---|---|
| B1 | ハイブリッド検索 | pgvectorのベクトル検索+`pg_trgm`/`tsvector`の全文検索を並列実行しマージ。固有名詞・完全一致の取りこぼしを防ぐ | 低 |
| B2 | エピソード記憶/意味記憶の分離 | `sigmaris_experience`(出来事)と`user_fact_items`(恒久的事実)の役割を明確化。週次で経験記憶を意味記憶に統合するジョブを追加(Reflexion的設計) | 中 |
| B3 | 記憶の自己検証ループ | `active_inquiry.py`を拡張し、確信度(confidence)が低い記憶を定期的に本人へ能動的に確認する | 中 |
| B4 | 記憶の出所(provenance)トラッキング | 各記憶(fact/decision/experience)が「どの会話・どのターンから生まれたか」を必ず参照できるようにする。Phase Cの自己診断精度評価の前提 | 低〜中 |
| B5 | 記憶の鮮度・矛盾ダッシュボード | `memory_validator.py`の減衰・矛盾検出結果をフロントエンドに可視化 | 低 |
| B6 | 話題遷移トラッキング | 現在の話題リストだけでなく、話題がどう推移してきたかの順序情報を保持 | 中 |

### B系(公開数値の最大化を狙う追加ロジック)

| # | 機能 | 概要 | 工数目安 |
|---|---|---|---|
| B7 | マルチホップ質問の分解検索 | 複雑な質問をサブクエリに分解し個別検索後に統合。多段階推論スコアに直結 | 中 |
| B8 | 時間認識型リランキング | 各記憶に有効期間を持たせ、意味的類似度×時間的近さ・有効性で再ランキング。時系列推論スコアに直結 | 中 |
| B9 | ナレッジグラフ層 | エンティティ(人・場所・プロジェクト等)と関係性をグラフ構造で保持。多段階推論の連鎖質問に強くなる | 高(最後に回す) |
| B10 | クロスエンコーダ二段階リランキング | 一次検索(top20)をクロスエンコーダで再採点しtop5に絞る。rag_ndcg_scoreに直結 | 高 |
| B11 | 校正された放棄判定 | 質問カテゴリごとに確信度閾値を学習的に調整。「わからない」を正しく答える精度に直結 | 中 |
| B12 | 検索後の圧縮(Post-retrieval Compression) | 検索結果をそのまま注入せず、質問に必要な部分のみLLMで抽出・圧縮してから注入。トークンコスト削減 | 中 |

### B系(パーソナル性能の最大化を狙う追加ロジック)

| # | 機能 | 概要 | 工数目安 | 優先度 |
|---|---|---|---|---|
| B13 | 暗黙フィードバックによる検索ランキング個人最適化 | 提案の採用/不採用ログを検索スコアのランキング関数にフィードバック。海星さん固有の重要度判断に検索が近づいていく | 中(B10と組み合わせ可) | 中 |
| B14 | **意思決定パターンのモデリング** | decision_logの蓄積から「海星さんはコストより速度を優先する」「OSSを好む」「長期保守性を優先する」等の判断傾向を抽出・保持し、新規提案に反映。**事実(fact)ではなく判断軸(preference structure)を学習する点がMem0にもほぼ存在しない差別化要素**。LoCoMo/LongMemEvalのような事実想起ベンチマークでは点数化されないが、パーソナルAIとしての核心的価値になる | 中〜高 | **高(引き上げ)** |
| B15 | 個人別放棄閾値 | 「間違うくらいなら黙る」か「仮説でも出してほしい」かを過去のやり取りから学習し、確信度閾値を個人向けに調整 | 中 | 中 |
| B16 | 長期ゴール整合性チェック | シグマリス計画の最終目標・海星さんが語る個人目標を別途追跡し、日々の提案がそれと整合しているかを定期チェック | 中 | 中 |
| B17 | **Memory Importance Learning(既存importance_scoreの活用範囲拡張)** | `user_fact_items`には既に`importance_score`列が存在し`build_facts_context()`で検索抽出に使われている。これを**新設ではなく、要約されにくさ・忘却されにくさ・Retrieval順位への反映まで一貫させる改修**として実施する。重要度が高い記憶(例:「シグマリスを作る」)ほど圧縮・減衰の対象から外れ、検索順位でも優先されるようにする。`memory_validator.py`の減衰ロジックとの統合が必要 | 中 | 高 |

**着手順の目安(v2: 小さなPDCAで反復)**:

```
B1(ハイブリッド検索) → C-miniで評価
B4(出所トラッキング) → C-miniで評価
B17(importance学習の一貫適用) → C-miniで評価
B14(意思決定パターン) → C-miniで評価  ← 差別化の核。B13と合わせて重点的に
B13(暗黙フィードバック) → C-miniで評価
B2・B3・B6(エピソード分離・自己検証・話題遷移) → C-miniで評価
B7・B8(マルチホップ分解・時間認識リランキング) → C-miniで評価
B10・B11・B12(クロスエンコーダ・放棄判定・圧縮) → C-miniで評価
B15・B16(個人別放棄閾値・長期ゴール整合性) → 余力があれば
B9(ナレッジグラフ) → 「本当に必要になったら」。RAG+ハイブリッド検索+クロスエンコーダだけでも
                        性能はかなり伸びる見込みのため、工数対効果で最後に回す
```

## Phase C: 測定基盤

**目的**: 独自指標と公開ベンチマークの両方で客観的に性能を可視化する。**Phase D以降に進む前に必ず完成させる。**

### C1. シグマリス独自指標(公開・is_public=true)

引き継ぎ文書の元設計をそのまま採用。

- SB-1 memory_f1_score(目標0.85)
- SB-2 rag_ndcg_score(目標0.70)
- SB-3 memory_duplicate_rate(目標3%以下)
- SB-4 response_error_rate(目標1%以下)
- SB-5 curiosity_relevance_score(目標0.70)
- SB-6 self_diagnosis_accuracy(目標70%)
- SB-7 improvement_cycle_gain(目標5%以上)

### C2. 公開ベンチマークの流用(自作しない)

- **LongMemEval**: 公開されている会話履歴+500問のQ&Aセットをシグマリスの記憶パイプライン(抽出→保存→検索→応答)にそのまま流し込み、標準のLLM-as-a-Judge方式で採点する
- **LoCoMo**: 同様に公開データセット(1,540問)をそのまま利用する
- 自作Q&A生成(バイアスリスクがある)は主軸にしない。個人特化部分の補助評価としてのみ将来的に検討する
- 可能であればMem0(OSS版)をローカルで並行稼働させ、同一のLongMemEval/LoCoMoデータで**paired比較**を行う。これが最も反論されにくい「上回った」の根拠になる

### C3. 目標スコアライン(LoCoMo基準)

| スコア帯 | 位置づけ |
|---|---|
| 〜55% | 素の実装(OpenAI標準メモリ相当) |
| 60〜70% | **最低目標ライン**。Phase A完了時点でほぼ到達見込み |
| 70〜75% | **現実的な最終目標**。Phase B一式+チューニング数サイクル |
| 75〜85% | ストレッチ目標。届けば個人開発として胸を張れる水準 |
| 90%以上 | 非目標。専任チーム相当の継続開発が必要で個人開発の射程外 |

カテゴリ別(単発/多段階/時系列/知識更新/放棄)の内訳も必ず記録し、総合点だけでなく弱点カテゴリを把握する。

### C4. ダッシュボード反映

B5(鮮度・矛盾ダッシュボード)とC1〜C3の指標を統合表示。改良サイクルごとのスコア推移を可視化する。

## Phase D〜H: 自己改良システム本体(旧タスク3 Phase2〜6)

Phase C-mini/C-fullが完成し、効果測定が可能になった後に着手。

- **Phase D**: 改良提案エンジン(根拠収集・Pushover承認)
- **Phase E**: 自動テスト環境(Docker・効果測定・自動ロールバック)★最重要。ここが実際に動くことが「業界的に個人でここまでやったか」と言われる最大の要素
- **Phase F**: 自動実装パイプライン
- **Phase G**: フロントエンド成長ログ
- **Phase H**: X投稿連携(完成後、根拠データ付きで投稿)

### 自己改良ループの詳細設計(重要な安全設計を含む)

**前提となるリスク認識**: 自分自身の出力を自分自身が評価して改良し続けるループは、グッドハートの法則(測定基準が目標化すると、その基準は良い基準でなくなる)のリスクを常に抱える。内部指標(SB-1〜7)は自己言及的(評価ロジック自体がシグマリスの一部)なため、単独では改良の正しさを保証できない。

**対策: 二層評価によるハッキング検知**

```
① トリガー(週次予算選択・閾値割れ検知)
② 根拠収集(失敗ログ・decision_log・論文/GitHub参照)
③ 改善仮説の生成(LLMによる提案+根拠の明文化)
④ サンドボックス実装(Docker・本番非接続)
⑤ 効果測定(二層)
   a. 内部指標(C-miniの3指標+SB-1〜7) ← 高速・頻繁な一次判定
   b. 外部ベンチマーク(LongMemEval/LoCoMo) ← 低頻度(例: 月次)・信頼性の担保
⑥ 判定
   - 内部指標↑ かつ 外部ベンチマーク非悪化 → 昇格候補
   - 内部指標↑ だが 外部ベンチマーク↓ → 却下。グッドハート兆候として記録・分析対象にする
⑦ Pushover通知 → 海星さんが1タップ承認
⑧ 本番適用
⑨ 継続監視。悪化検知で自動ロールバック
```

**重要な設計原則**: 内部指標だけが上がり続け、外部ベンチマーク(LongMemEval/LoCoMo)が横ばい・悪化している状態が続く場合、それは改良ループが評価基準をハックしている兆候として扱い、改良提案エンジン自体の見直し(仮説生成プロンプトの再設計等)を優先する。外部ベンチマークは頻繁に回す必要はないが、内部指標の信頼性を担保する「定期監査」として必ず組み込む。

## 「業界的に個人でここまでやったか」の訴求ポイント(ベンチマーク数値以外)

ベンチマーク絶対値の最大化(90%以上)は非現実的なため、以下を主軸に据える。

1. **クローズドループが実際に週次で回っている実績**(Phase D〜Eの完全動作) — 資金調達したチームでも中途半端なことが多く、最もインパクトが大きい
2. **再現可能な形での公開**(評価スクリプト・スコア推移ログをOSS化) — 数値そのものより検証可能性が信頼を生む
3. **リソース制約下での効率性**(GTX 1660 6GB・週次予算制という制約下での成果) — クラウド依存の大規模記憶システムとの差別化軸
4. **MOSSと同種のソースコードレベル自己書き換えの実運用実証** — 研究段階のものが多い領域での実例
5. **単一ユーザーへの深い個人適応のケーススタディ**(定性面) — 汎用ベンチマークが測らない領域の補完

## 着手順の結論(v2)

```
Phase A(A0→A1→A2→A3→A4→A5)
   ↓
Phase C-mini(memory_f1 / rag_ndcg / response_error_rateの3指標のみ先に用意)
   ↓
B1→B4→B17→B14→B13→B2/B3/B6→B7/B8→B10/B11/B12→(B15/B16→B9)
   ※ 1機能実装するごとにC-miniで評価し、次に進む(小さなPDCA)
   ↓
Phase C-full(独自指標SB-1〜7全部 + LongMemEval/LoCoMo公開ベンチマーク運用)
   ↓
Phase D〜H(自己改良システム本体)
```

**外部レビュー(ChatGPT)を踏まえた変更履歴**:
- v1→v2: Phase Cを丸ごと先に完成させる設計から、最小限の評価基盤(C-mini)だけ先に作り、B群を1つずつ実装・評価する小さなPDCAループに変更
- B14(意思決定パターンのモデリング)の優先度を中〜高から「高」に引き上げ。事実想起ではなく判断軸の学習であり、ベンチマークには表れないがMem0にもない差別化価値と判断
- B17(Memory Importance Learning)を追加。ただし既存の`user_fact_items.importance_score`と重複するため、新設ではなく「既存列を検索順位・忘却耐性まで一貫して反映させる改修」として位置づけ
- B9(ナレッジグラフ)は当初案から変わらず最後に据え置き。工数対効果の観点で「本当に必要になったら」の位置づけを維持

次のアクション: Phase A0(書き込み経路の一本化)のClaude Codeへの実装指示文を作成する。
