## 1. エンティティ・関係テーブルの設計詳細

### 最小構成(グラフDB不使用)

指示書の最優先制約に従い、専用グラフDB・グラフクエリ言語は一切導入していない。
既存のSupabase/PostgreSQL上に、通常のJOINで辿れる2テーブルのみを追加した:

```sql
sigmaris_entities (id, name, entity_type, created_at)
  -- entity_type: person / place / project / technology / other
  -- unique(name, entity_type)

sigmaris_entity_relations (
  id, from_entity_id, to_entity_id, relation_type,
  source_table, source_id, created_at
)
  -- unique(from_entity_id, to_entity_id, relation_type)
```

`entity_type`は指示書が例示した「人物・場所・プロジェクト・技術要素」に、汎用の
`other`を加えた5種類のみに固定した。関係の種類(`relation_type`)は固定enumにせ
ず自由記述の短い文字列とした(判断根拠): 関係性は「取り組んでいる」「の一部で
ある」等、多様かつ事前に列挙しきれないため、固定enumにすると早々に破綻すると判
断した。エンティティ種別は5種と少なく網羅的に列挙できるため固定enum、関係種別は
自由記述、という非対称な設計にした。

### B16の申し送りを踏まえた設計の一貫性

`docs/sigmaris/phase_b16_report.md`が指摘した「B2・B6・B14・B15・B16が独立して
同じパイプライン構造(週次抽出→グローバルなservice-roleテーブル→TTLキャッシ
ュ注入)を実装してしまった」という問題を踏まえ、本タスクでは**B16の
`goal_alignment.py`の構造をそのまま踏襲**した(判断根拠): 6つ目の独立した実装
パターンを生み出さないという指示書の明示的な制約に従い、既存5パターンのうち最
も直近かつ本タスクの読み込み対象(goals・decision_log・topic_log)が完全に一致
するB16を選んだ。`_svc_headers`のヘルパー関数構造、`get_or_create`パターン(た
だしB14/B16の証拠蓄積パターンとは異なる、2章で詳述)、週次ジョブの登録方法まで
一貫させた。

## 2. 既存資産との重複回避の方法

### 生データを複製しない設計

`sigmaris_entities`は名前と種別のみを保持し、決定内容・事実の値・話題の詳細等は
一切保持しない。`sigmaris_entity_relations`の`source_table`/`source_id`は、関係
がどの既存レコード(`sigmaris_decision_log`または`sigmaris_topic_log`)から抽出
されたかを**指し示すだけ**で、その内容を複製しない。抽出LLMへの指示文
(`_EXTRACT_PROMPT`)にも「決定の詳細な理由や内容そのものを複製しないこと」を明
記した。

### B14/B16と異なり、証拠件数によるゲーティングを行わなかった(判断根拠)

B14(`_MIN_SUPPORTING_DECISIONS`)・B16(`_MIN_SUPPORTING_EVIDENCE`)は、いずれ
も「複数の証拠がないと判断そのものを保存しない」というゲーティングを行ってい
た。本タスクではこれを**あえて行わなかった**: B14・B16が扱うのは「海星さんの判
断傾向」「目標との乖離」という、**断定すると誤りのコストが高い主観的判断**であ
るのに対し、本タスクが扱うのは「AというエンティティとBというエンティティの間に
何らかの関係が言及された」という、より軽量で訂正コストの低い事実の記録である。
根拠(`source_ref`)が検証できない場合でも、関係自体は保存し、出所情報だけが
`null`になる設計にした(単体テストで直接確認、4章)。これは「薄い補完レイヤー」
という位置づけに沿った、意図的な非対称設計である。

### `get_or_create`による冪等性(重複防止)

`sigmaris_entities`の`unique(name, entity_type)`制約と、`_get_or_create_
entity()`の「まず検索し、なければ作成する」ロジックにより、同じエンティティが
複数回抽出されても行が重複しない。関係も同様に`unique(from_entity_id,
to_entity_id, relation_type)`で重複を防ぐ。B14/B16の証拠蓄積(`evidence_refs`を
マージする)とは異なり、本タスクでは同じ関係が再度抽出された場合は単に既存行を
再利用するだけで、追加の蓄積ロジックは持たない(過剰実装を避けるための判断)。

## 3. B7との連携方法

### 新規LLM呼び出しゼロでの統合

B7の`multihop_search.decompose_query()`は、既にB6の話題ラベルを「参考ヒント」
として同じプロンプトに追加する仕組み(`_format_topic_hint`)を持っていた。本タ
スクはこれと**全く同じ機構**を再利用し、`_format_entity_hint`を追加しただけであ
る。

- `knowledge_graph.build_entity_hint(entities, relations, query_text)`は純粋な
  Python関数であり、LLM呼び出しもI/Oも一切行わない。クエリ文字列に含まれるエン
  ティティ名を単純な部分文字列一致で検出し、そのエンティティに関連する関係を最
  大3件(`_MAX_HINT_RELATIONS`)まで短いテキストにまとめるだけである。
- `orchestrator/service.py`は、既存のTTLキャッシュ機構(`_cached_preference_
  patterns`等と同じパターン)で全エンティティ・関係のスナップショットを300秒キ
  ャッシュし(`_cached_entities_and_relations`)、このキャッシュ済みデータに対し
  て`build_entity_hint`を呼ぶ。応答経路上で新たなDB呼び出し・LLM呼び出しは一切発
  生しない(キャッシュヒット時)。
- `decompose_query()`のプロンプトに`entity_hint`を追加のヒントブロックとして挿
  入するだけで、B7自体の呼び出し回数・構造は一切変更していない。

## 4. テスト結果

実モデルAPIでの検証はできないため、mock/単体テストで確認した(2ファイル、計12
ケース、これまでのB群と同じくスクラッチテストとして作成・実行、backend/tests
配下にはコミットしていない)。

### `extract_entities_and_relations()`(5ケース)
- 決定件数が最低件数未満の場合、LLMを呼ばずinsufficient_dataを返すこと
- 検証済みの根拠を持つ有効な候補が正しく保存され、`source_table`/`source_id`が
  正しく設定されること
- 無効な`entity_type`を持つ候補が(エンティティ作成すら試みずに)棄却されること
- **根拠が検証できない候補でも、関係自体は保存され、出所情報のみ`null`になる**
  こと(2章の判断根拠の直接確認、B14/B16との明確な違い)
- LLM呼び出し失敗時に例外を握りつぶすこと

### `build_entity_hint()`(5ケース)
- エンティティ・関係が空の場合`None`を返すこと
- クエリにエンティティ名が含まれない場合`None`を返すこと
- クエリにエンティティ名が含まれる場合、関連する関係を含むヒントが生成されるこ
  と(定性的なマルチホップ改善効果の直接確認: 「AdFlow AIについて教えて」という
  質問に対し、「AdFlow AI —目標にしている— 収益化」という関連情報が表面化する)
- マッチしたエンティティに関係が一切ない場合`None`を返すこと
- ヒントに含まれる関係数が上限(3件)を超えないこと

### `orchestrator/service.py`の配線(2ケース)
- `_build_memory_context()`が正しいエンティティ・関係・クエリ文字列で
  `build_entity_hint()`を呼び、その結果が`search_with_decomposition()`に
  `entity_hint`として渡されること
- `entities`/`relations`引数が渡されない場合(デフォルト`None`)でも例外を投げ
  ないこと

### 既存機能への非破壊確認(要件5)
`backend/tests/`の既存8テストを含む、これまでの全B群スクラッチテストを合わせた
193ケースを再実行し、全て成功することを確認した。

```

181 passed (B2/B3/B6/B7/B8/B10/B11/B12/B15/B16スクラッチテスト・facts cache
            修正テスト・既存回帰テスト再実行)
```

### 週次ジョブのスケジュール

既存ジョブ(`narrative_generate` 5:00・`self_interest_queries` 5:30)の間に30分
の空きがあることを確認し、`knowledge_graph_extract`を**日曜5:15**に配置した(既
存ジョブといずれも15分の間隔)。

## 5. Phase B全体(B1〜B17)を通じての振り返り

**注記**: 本タスクの依頼文は「Phase B(全17機能)完了」と記載しているが、ロード
マップ(`sigmaris_roadmap.md`)を確認したところ、**B5(記憶の鮮度・矛盾ダッシュ
ボード、フロントエンド可視化機能)は本セッションの実行順序(`B1→B4→B17→B14→
B13→B2/B3/B6→B7/B8→B10/B11/B12→(B15/B16→B9)`)に含まれておらず、未着手のまま
残っている**。B5はバックエンドのメモリパイプラインではなくフロントエンドの可視
化タスクであり、性質が異なるため意図的に別枠にされていた可能性が高いが、「全
17機能完了」と称するのは正確ではないため、ここで明記しておく。本セッションで完
了したのはB1〜B4・B6〜B17の16機能である。

### 技術的な教訓

1. **「複数の重み付けを合成する機能」は、必ず実測で境界を検証する必要がある**
   (B13・B8・B15で確立): RRFスコアは隣接ランク間の差が小さいため、一見小さな重
   み(0.05〜0.15程度)でも合成すると想定以上にランキングを動かしうる。B8では「基
   準値自体がB13の報告書の数値と異なる」という発見もあり、テスト設定の違いが数
   値比較を難しくすることも分かった。
2. **応答経路上のレイテンシは、B7で最初に導入されて以降、累積的に管理する必要が
   ある**: B7(分解判定)・B10(リランキング)の2つが応答経路上に新規LLM呼び出
   しを追加し、B11・B12・B15・B16・B9はいずれもこの制約を厳格に守って新規呼び出
   しを追加しなかった。この「予算」を意識した設計判断がB11以降のタスクで一貫し
   て機能したことは、この種の制約が後続タスクに正しく申し送りされた良い例だと考
   えられる。
3. **fire-and-forgetの「保留状態→後続ターンでの消費」というパターンは、B3で確
   立されて以降、B15(スレッド単位)・B16(グローバル)で異なるスコープで再利用
   された**: このパターン自体は非常に汎用性が高く、今後の機能でも第一候補として
   検討する価値がある。
4. **`importance_score`(B17)が実は粗いカテゴリ代理指標だったという発見は、B12
   (圧縮)・B16(目標整合性)・B9(本タスク)に一貫して影響を与えた**: 「重要
   度」を扱う機能を実装する際は、それが本当に個別の学習値なのか、カテゴリ固定値
   の代理なのかを都度確認する必要があるという教訓が、複数回のタスクを通じて実証
   された。
5. **CRLF/LFの差分ノイズ、mockの`self`バインディングの罠など、細かいが繰り返し
   遭遇した実装上の落とし穴がある**: これらは`git diff --ignore-all-space`での
   確認、mock関数への`self`引数追加、という具体的な対処法が確立され、後続タスク
   でも同じ問題への対処時間が短縮された。

### 残っている懸念事項

1. **5(本タスクを含めれば6)つの類似パイプラインの重複**: `docs/sigmaris/
   phase_b16_report.md`で指摘された、B2・B6・B14・B15・B16(+本タスクB9)が同型
   の「週次抽出→グローバルservice-roleテーブル→TTLキャッシュ注入」パイプライン
   をそれぞれ独立に実装している状態は、本タスクでも(指示書の指示通り)解消して
   いない。共通化するリファクタリングは、指示書が明示した通り本タスクの範囲外
   であり、**Phase C-full着手前、または別途のリファクタリングタスクとして改めて
   検討することを強く推奨する**。共通化の候補としては、「週次抽出ジョブの雛形」
   「service-role専用テーブルへのget-or-create/upsertヘルパー」「TTLキャッシュ
   付き注入ヘルパー」の3点が考えられる。
2. **実モデルAPIでの効果測定が一度も行われていない**: 本セッションを通じて、
   `OPENAI_API_KEY`・SSH/サーバーアクセスのいずれも一切得られなかったため、B1〜
   B17の全機能が、mock/単体テストによる論理的な検証のみで実装されている。実際の
   `rag_ndcg_score`・`memory_f1_score`・`response_error_rate`(C-miniで整備済
   み)への効果は、運用者側での実行を待つ状態が続いている。
3. **13件の未適用マイグレーション**: 本セッションで作成した以下のマイグレーシ
   ョンは、いずれも運用者側での適用が必要な状態のまま残っている:
   `202607060028`(C-mini)・`202607070029`(B1)・`202607080030`(B4)・
   `202607090031`(B17)・`202607100032`(B14)・`202607110033`(B13)・
   `202607120034`(B2)・`202607130035`(B3)・`202607140036`(B6)・
   `202607150037`(B8)・`202607160038`(B15)・`202607170039`(B16)・
   `202607180040`(B9、本タスク)。適用順序に依存関係があるため(例:B13のRPCは
   B17のRPC定義の上に構築されている)、**作成された順序通りに適用することを推奨
   する**。
4. **`sigmaris_decision_log`の実際のデータ蓄積量が本セッションを通じて一度も確
   認できていない**: C-miniのbaseline取得以降、B13〜B16の複数機能が
   `sigmaris_decision_log`のデータ量に依存する形で設計されているが(最低証拠件
   数のゲーティング等)、実際の行数は運用者側でのみ確認可能であり、これらの機能
   が実運用でどの程度「insufficient_data」に倒れているかは未確認である。
5. **`/chat`表示崩れの調査(本セッション序盤)が未解決のまま残っている**: B群タ
   スクとは別系統の調査依頼だったが、ユーザーからの追加指示がないまま保留になっ
   ている。

### Phase C-full着手前に対応しておくべき事項

1. **上記13件のマイグレーションを全て適用すること**(最優先): これが行われな
   い限り、B群の実装コードの大部分(RPCの新しい戻り値列、新規テーブルへの読み書
   き)は本番環境で実際には動作しない。
2. **`run_eval.py`を実際のマイグレーション適用後・実モデルAPI環境で実行し、
   C-miniの3指標の実測値を取得すること**: 本セッション中に記録された「参考値」
   は全て、同日中の複数の修正が混在した状態、またはデータ不足の状態で取得された
   ものであり、真の意味でのB群機能の効果測定はまだ行われていない。
3. **5パイプラインの重複を解消するリファクタリングを検討すること**(2点目の懸
   念事項): 直接Phase Cの機能追加を妨げるものではないが、放置すればB群の資産を
   横断的に活用する将来の機能(本タスクのようなもの)のたびに、一貫性を保つ判断
   コストが積み重なる。
4. **応答経路上のLLM呼び出し数(現状B7・B10の2つ)の実測レイテンシを確認するこ
   と**: B7の報告書で見積もりとして記載した数値(平均300〜1,200ms程度)は実測で
   はなく、実際のユーザー体験への影響は運用データでの確認が必要。
5. **B5(記憶の鮮度・矛盾ダッシュボード)の着手方針を決定すること**: 本セッショ
   ンのスコープには含まれていなかったフロントエンド機能であり、Phase C以降でど
   う扱うかを別途決定する必要がある。
