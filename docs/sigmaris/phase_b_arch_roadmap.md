# Phase B-arch: アーキテクチャ再設計ロードマップ(割り込みフェーズ)

## 位置づけ

```
Phase A(完了)→ Phase B(完了、B1〜B17)
   ↓
【新設】Phase B-arch ← Phase C-fullに割り込んで先に実施
   ↓
Phase C-full(測定基盤の本格運用)
   ↓
Phase D〜H(自己改良システム本体)
```

## 背景

Phase B完了後の実測で、1回の応答に約36秒かかることが判明した。原因調査の結果、最大の要因は「実務応答生成→人格変換」という**二段階の完全に別のLLM生成**であり(全体の60%以上)、次いで記憶検索まわりの重複・無駄な取得が続く。

これを受け、外部レビュー(ChatGPT)から「Hot Path(会話中)/Cold Path(会話外)の分離」「人格AIの常駐化」「Memory Snapshot方式」という設計案が示された。学術的にも、認知アーキテクチャ・オフライン学習/オンライン推論分離・Reflexion由来のエピソード記憶・VeriFact-CoT/SAVER由来の自己検証内蔵型生成、という複数の裏付けがある方向性であることを確認済み。

本フェーズは、B1〜B17の内部ロジックは変更せず、**「いつ・どこで呼ばれるか」という配線と、応答生成そのものの構造**を再設計する。

---

## サブフェーズ構成

### BA1: Hot Path上の残存処理の退避(最も低リスク)

**現状**: B2・B6・B9・B14・B16は既にfire-and-forget(Cold Path)。**B3(記憶の自己検証)のみ、応答経路上(Hot Path)に残っている**上、タイムアウト(2秒)ぎりぎりで動作し、サイレント失敗の疑いがある。

**やること**:
- B3の確認質問生成を、応答をブロックしない完全な非同期処理に変更する(他のB群機能と同じfire-and-forgetパターンに揃える)
- タイムアウト付近での失敗有無を実際に調査し、原因を特定・修正する
- 確認質問は「今回の応答には即座に含めず、次のターン以降に自然に織り込む」設計に変更する

**期待効果**: 最大2秒の短縮+隠れていた不具合の解消。低リスクで着手しやすい。

---

### BA2: 重複・無駄な取得の除去(低リスク、着手済みのものと統合)

**現状**: `get_profile_context()`が3箇所で重複取得。`user_fact_items`取得時に不要な768次元embeddingまで毎回取得。

**やること**:
- `get_profile_context()`の結果をリクエスト単位でキャッシュし、3回の呼び出しを1回に集約する
- 記憶一覧取得時、embeddingベクトルが不要な経路では`select`を絞り込む
- 既に指示済みの「facts_ctx/trends_ctxの握りつぶしバグ」修正もこのタイミングで統合する

**期待効果**: 1〜2秒程度の短縮。BA1と合わせて、まず着実な土台を作る。

---

### BA3: Memory Snapshot方式の導入

**現状**: 記憶(383件)を毎回全件取得し、その場で検索・重み付けしている。B14(判断傾向)・B16(目標整合性)・B6(話題)・B9(関係性)の結果は、個別にDBへ読みに行っている。

**やること**:
- B14・B16・B6・B9の出力を、**週次の裏処理(Cold Path)で1つの「現在のユーザー像」(Snapshot)にまとめる**新しい集約処理を追加する
- 会話時は、まずこのSnapshotを読み込み、そこに含まれない詳細が必要な場合のみ、B1のハイブリッド検索で個別に補う
- Snapshotの生成は、既に確立された5〜6パイプラインの重複解消(Phase B完了時の申し送り事項)と合わせて設計する。**ここでB2・B6・B14・B16・B9の重複パイプラインを、Snapshot生成という1つの共通処理に統合する**ことで、技術的負債の解消と速度改善を同時に行う

**期待効果**: 記憶読み込み・重み付け計算の大部分が、会話中ではなく週次処理側に移動する。加えて、長年の懸案だった「5〜6パイプラインの重複」も同時に解消される。

---

### BA4: 応答生成の統合(最重要・最難関)

**現状**: `chat.py`(実務応答生成)→`persona_rewriter.py`(人格変換+事実確認)という、完全に独立した二段階のLLM生成。

**やること**:
- 実務処理(ツール呼び出し等)と人格表現を、**最初から1つの生成プロセスで行う**よう再設計する
- 事実確認の役割(`compare_semantic_entities()`が担っていたもの)は、VeriFact-CoT/SAVERの考え方を参考に、**「全部を書き直して照合する」のではなく「生成直後に短い自己点検を行い、問題があれば該当箇所のみ修正する」**という軽量な仕組みに置き換える
- ツール呼び出し(カレンダー登録等)の安全性(確認ボタンの仕組み等、Phase A1-bで保護した部分)は、統合後も同様に機能することを慎重に確認する
- この変更は影響範囲が最も大きいため、**BA1〜BA3が完了し、実際に速度改善が確認できた状態**で着手する

**期待効果**: 二段階生成が一段階になることで、全体の応答時間から大幅な削減(理論上、全体の6割を占めていた区間の相当部分)が見込める。

---

### BA5: 実測・検証

**やること**:
- BA1〜BA4それぞれの完了後、都度サーバー上で実際の応答時間を実測する(推定に頼らない)
- 今回発見した「36秒」との比較を、各段階で記録する
- C-miniのスコアも都度確認し、速度改善によって記憶精度が犠牲になっていないことを確認する

## 実装後、応答はどう変わるか(イメージ)

### 現状(Before)

```
海星さん「明日の予定教えて」
   ↓ (383件の記憶取得、複数箇所での重複読み込み)
   ↓ (実務AIがカレンダーを調べて回答文を作成、約12秒)
   ↓ (人格AIがゼロから書き直し、約10秒)
   ↓ (その他の処理)
シグマリス「明日は10時から会議がありますよ」
   ※ここまで約36秒
```

### BA1〜BA3完了後

```
海星さん「明日の予定教えて」
   ↓ (Snapshotを読み込む。判断傾向・目標・話題は既に整理済み)
   ↓ (カレンダーの詳細だけ、その場でピンポイント検索)
   ↓ (実務AI→人格AIの二段階生成は残っている)
シグマリス「明日は10時から会議がありますよ」
   ※記憶まわりの数秒が短縮されるが、二段階生成が残るため
   　全体としてはまだ十数秒〜20秒台の見込み
```

### BA4完了後(最終形)

```
海星さん「明日の予定教えて」
   ↓ (Snapshotを読み込む)
   ↓ (カレンダーの詳細をピンポイント検索)
   ↓ (シグマリス本人が、実務も口調も一体で直接生成。
      生成直後に短い自己点検のみ)
シグマリス「明日は10時から会議がありますよ」
   ※理論上、数秒程度まで短縮される見込み
```

### 裏側(会話に見えない部分)で常時起きていること

```
【毎晩・毎週】
- その日の会話からエピソードを整理(B2)
- 判断傾向を確認・更新(B14)
- 長期目標との整合性をチェック(B16)
- 話題の推移・関係性を整理(B6・B9)
- これら全てを1つの「現在のあなたについての理解」(Snapshot)にまとめる(BA3)

【会話中】
- Snapshotを読み込む(軽い)
- 直近の会話・今必要な詳細だけ、その場で検索する(Phase A1・B1はそのまま活用)
- 実務も人格も一体で、その場で直接答える(BA4)
- 応答直後、短い自己点検だけ行う
- 会話が終わったら、その内容もまた次の夜の整理に回る(循環)
```

- 各BAサブフェーズの指示書は、これまでのPhase A・B同様、1つずつ独立して作成し、実測を挟みながら進める
- BA3で5〜6パイプラインの重複解消を行うため、Phase B完了時の申し送り事項(`phase_b_summary.md`4章)は、本フェーズ完了をもって解消される見込み
