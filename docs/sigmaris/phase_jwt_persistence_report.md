# 緊急対応 実施報告: Supabase JWTリフレッシュトークンの永続化

**目的:** `jwt_manager.py`のリフレッシュトークンローテーションがプロセスのメモリ上にしか保持されず、常駐プロセス再起動時に`.env`の(既に使用済みの)値でのブートストラップに失敗し`refresh_token_already_used`を起こす問題を解消する。
**作業ブランチ:** `phase-jwt-persistence`(`main`から新規作成)
**マージ状態:** **未マージ。このタスク専用ルールにより、報告・承認を待ってマージする。**
**現在有効なセッションへの影響:** なし。実装・テストは全て新規のモック/一時ファイルに対して行っており、本番の`.env`・常駐プロセス・現在有効なリフレッシュトークンには一切アクセス・変更していない。

---

## 1. 永続化先として何を選んだか、その判断根拠

**リポジトリ内のローカルJSON状態ファイル(`backend/.state/sigmaris_jwt_session.json`)を選んだ。`.env`への書き戻し・専用DBテーブルのどちらも採らなかった。**

### `.env`書き戻しを採らなかった理由

1. `backend/app/config.py`を確認したところ、`settings = Settings()`はモジュールインポート時に**一度だけ**インスタンス化される(pydantic-settingsのシングルトン)。つまり`.env`は**プロセス起動時に1回しか読まれない**。稼働中のプロセスが自分自身で`.env`を書き換えても、当のプロセス自身はそれを再読込しない — 書き戻しの恩恵を受けるのは次回起動時のみであり、これは専用の状態ファイルを使った場合と全く同じ話になる。つまり`.env`書き戻しには「次回起動時に効く」という以上のメリットがない一方、以下のデメリットだけが残る。
2. `.env`は`SIGMARIS_REFRESH_TOKEN`以外にも多数の設定値(APIキー・シークレット等)を含む共有ファイルであり、パース→値だけ書き換え→再シリアライズという処理は、コメント・フォーマット・他の値を壊すリスクを伴う。専用ファイルなら影響範囲をこの1機能に閉じ込められる。
3. `.env`は既に`.gitignore`で`.env*`パターンにより保護されているが、意図せぬ手動編集・デプロイスクリプトによる上書きの対象にもなりうる。認証状態という「頻繁に自動更新される値」と、「人間が手で管理する設定値」を同じファイルに混在させたくなかった。

### DBテーブルを採らなかった理由

このトークンはSupabaseへの認証そのものに使うものであり、Supabase自体(またはそれに準ずるサービスロール権限)に依存する永続化先を選ぶと循環的な依存になりうる(DBへの書き込みにまた別の認証情報が必要になる、DB接続不可時にトークン更新自体が止まる等)。今回の問題の本質は「プロセス再起動をまたいで、ローカルなプロセス状態を引き継げていない」ことであり、それを解決するのに外部サービスへの依存を増やす必要はないと判断した。マイグレーション適用(このリポジトリでは手動適用が必要)というデプロイ上のステップが追加で発生する点もデメリットとして考慮した。

### ローカルJSON状態ファイルの設計

- 場所: `backend/.state/sigmaris_jwt_session.json`(`.env`と同じ`backend/`直下の隠しディレクトリ)
- 内容: `access_token`・`refresh_token`・`expires_at`(UNIXタイムスタンプ)・`persisted_at`(書き込み時刻、デバッグ用)
- 書き込みは一時ファイル(`.json.tmp`)に書いてから`Path.replace()`でアトミックにリネーム(書き込み途中のプロセスクラッシュによる破損ファイルを防ぐ)
- 可能な場合はPOSIXパーミッション`0600`を付与(Windows開発機では単に無視される — ベストエフォート)
- `.gitignore`に`/backend/.state/`を追加し、`.env`と同等の機密度としてコミット対象から除外した

## 2. 実装詳細(コード変更箇所の一覧)

対象ファイル1つのみ: `backend/app/services/proactive/jwt_manager.py`

| 変更 | 内容 |
|---|---|
| `_STATE_FILE`定数追加 | `backend/.state/sigmaris_jwt_session.json`への絶対パス。`jwt_manager.py`自身の場所から`parents[3]`で`backend/`を算出(動作確認済み、3章参照) |
| `_load_persisted_state()`新設 | 状態ファイルを読み、`_TokenState`に変換。ファイル不在・壊れたJSON・型不一致等、**あらゆる失敗を`None`として扱い例外を投げない**(呼び出し側は「何も永続化されていない」と同じ扱いで`.env`ブートストラップにフォールバックする) |
| `_persist_state()`新設 | 一時ファイル書き込み→chmod(ベストエフォート)→アトミックリネーム。**失敗しても例外を外に投げず、ログに記録するだけ**(要件2に対応、4章のテストで確認) |
| `get_sigmaris_jwt()`のブートストラップ優先順位変更 | `_state is None`の初回、**永続化ファイル→(なければ)`.env`の`SIGMARIS_USER_JWT`/`SIGMARIS_REFRESH_TOKEN`**の順で試す。既存の「`SIGMARIS_REFRESH_TOKEN`が全く設定されておらず、かつ永続化もない場合は静的JWTをそのまま返す(自動更新なし)」という元の挙動は、条件分岐に永続化チェックを追加しただけで**そのまま保持**した | 
| `get_sigmaris_jwt()`のリフレッシュ成功時 | `_state = new_state`の直後に`_persist_state(new_state)`を呼ぶ1行を追加しただけ |

ロジックフロー全体の変更は、既存の`_do_refresh()`(Supabaseとの通信部分)・`_decode_exp()`・`_is_fresh()`には一切手を入れていない。変更は「どこから初期状態を読むか」と「更新後どこに書くか」の2点に閉じている。

## 3. テスト結果(再起動シミュレーション含む)

全てモック(Supabase API呼び出しは`httpx.AsyncClient.post`をモック、状態ファイルは一時ディレクトリを使用)。**本番の`.env`・常駐プロセス・現在有効なセッションには一切触れていない。**

```
PASS: successful refresh persists the new (rotated) token pair to disk
PASS: simulated restart bootstraps from persisted state file, ignoring stale .env values,
      with zero network calls (access token still fresh)
PASS: persistence failure (unwritable path) does not break the in-memory refresh chain
PASS: no full token value appears across 3 log lines emitted during a refresh+persist cycle
PASS: pre-existing 'no SIGMARIS_REFRESH_TOKEN configured' escape hatch (static JWT, no refresh) preserved
```

各ケースの意味:

1. **要件1(再起動をまたいだ引き継ぎ)の核心**: リフレッシュ成功時に新しいペアがディスクに書かれることを確認。
2. **要件1の直接的な検証**: モジュール状態(`_state`・`_lock`)をリセットして「プロセス再起動」を模擬し、`.env`にはあえて「既に使用済みの古い値」を設定した状態で、**永続化ファイルから正しくブートストラップし、かつネットワークに一切アクセスしない**(アクセストークンがまだ有効期限内のため)ことを確認した。これはまさに今回の障害シナリオ(`.env`の値は死んでいるが、永続化された値は生きている)を再現したテストである。
3. **要件2**: 状態ファイルの親ディレクトリの代わりに通常ファイルを置くことで書き込みを意図的に失敗させ、それでも`get_sigmaris_jwt()`自体は正常にトークンを返し続け、以降の呼び出し(インメモリキャッシュのヒット)も壊れないことを確認した。
4. **要件3**: ログハンドラーを差し込み、リフレッシュ〜永続化の一連の処理で出力される全ログ行に、実際のトークン値(テストで使った秘密のマーカー文字列)が一切含まれないことを確認した(既存コードの「末尾6文字のみログ」という設計を維持し、新規追加したログ行もステータスのみ)。
5. **既存挙動の非破壊確認**: `SIGMARIS_REFRESH_TOKEN`が未設定・永続化ファイルも存在しない場合、元のコードと同じく「静的JWTをそのまま返す(自動更新なし)」という経路が変わっていないことを確認した。

### 既存テストスイート

`backend/tests/`(8件)全てPASS。`import app.main`成功。

### 要件4(既存の常駐プロセスへの悪影響がないこと)について

指示書にもある通り、本番の`sigmaris-backend`に対する再起動・セッション失効を伴うテストは行っていない(禁止事項として明記されていたため)。上記のモックテストに加えて、コードレベルで以下を確認した:

- 永続化ファイルが存在しない環境(=このコード変更を適用した直後、まだ一度もリフレッシュが起きていない状態)では、`_load_persisted_state()`は`None`を返し、**元の`.env`ブートストラップ経路にフォールバックする**。つまりデプロイ直後の常駐プロセスの挙動は、このコード変更前と完全に同じになる(4章で詳述)。
- 現在有効なリフレッシュトークンを消費・失効させるような操作(新規ログイン等)はコード中に一切含まれていない。

## 4. 本番の`sigmaris-backend`サービスへの適用手順(ダウンタイムなしでの切り替え)

**海星さんの承認後、実際にデプロイする場合の手順(私からは実行できないため、手順のみ提示する):**

1. このブランチ(`phase-jwt-persistence`)を`main`にマージ・pushする(承認後)。
2. サーバー側で`git pull`し、`sigmaris-backend`サービスを通常のデプロイ手順で再起動する。
3. **再起動直後は`.state/sigmaris_jwt_session.json`がまだ存在しないため、元の`.env`ブートストラップ経路が使われる**(3章「要件4」参照)。ここで`.env`の`SIGMARIS_REFRESH_TOKEN`が既に使用済みであれば、**この再起動時点で今回調査したのと同じ`refresh_token_already_used`エラーが1回だけ発生する**可能性が高い。これは今回のコード変更では解決されない — 別途、有効な新しいトークンペアで`.env`を更新してから再起動する必要がある(前回の調査報告の4章で述べた復旧手順が引き続き必要)。
4. **重要**: したがって、このデプロイは「壊れた`.env`の復旧」と**同時に**行うことを推奨する。具体的には:
   a. 有効な新しいアクセストークン・リフレッシュトークンのペアを取得する(前回報告の4章参照)
   b. `.env`の`SIGMARIS_USER_JWT`・`SIGMARIS_REFRESH_TOKEN`をこの新しい値に更新する
   c. このブランチのコードをデプロイし、サービスを再起動する
   d. 再起動後、初回の`get_sigmaris_jwt()`呼び出しで新しい`.env`の値からブートストラップし、以降のローテーションから`.state/sigmaris_jwt_session.json`への永続化が始まる
5. **ダウンタイムについて**: この変更自体はJWT取得ロジックのみの変更であり、サービス全体の起動シーケンスや他のエンドポイントには影響しない。通常のデプロイ(プロセス再起動)と同じダウンタイム(再起動に要する数秒程度)以上のものは発生しない。ゼロダウンタイムのブルーグリーンデプロイ等をこのシステムが採用していない前提であれば、追加の考慮は不要。
6. デプロイ後の確認: バックエンドのログで`"Sigmaris JWT bootstrapped from"`(初回)・`"Sigmaris JWT refreshed"`(以降のローテーション)・`"jwt_manager: persisted refreshed token state to disk (ok)"`(永続化成功)が出ていることを確認する。

## 5. 気づいた懸念点

1. **今回の変更だけでは、現在`.env`に保存されている値が既に使用済みである問題そのものは解決しない**(4章)。このタスクは「次に再起動しても壊れないようにする」ための恒久対策であり、「今すでに壊れているかもしれない`.env`の値」を直すには、別途トークンの再発行が必要(前回報告で提示済み、未実施)。
2. **`backend/.state/`ディレクトリのバックアップ**: このファイルが何らかの理由で失われると(ディスク障害等)、次回再起動時に`.env`の(古い)値へのフォールバックが発生し、今回と同じ問題が再発する。運用上、このファイルは`.env`と同程度に「失うと困るが、失っても`.env`さえ更新すれば復旧できる」性質のものとして扱うのが妥当。
3. **複数の常駐プロセス/ワーカーが将来増えた場合のリスク**: 現状はuvicornが単一プロセスで動いている前提(スケジューラーも同一プロセス内、Phase JWT調査報告3章で確認済み)。もし将来ワーカー数を増やす・別プロセスでスケジューラーを分離する等の変更があった場合、複数プロセスが同じ状態ファイルに同時に書き込もうとして競合する可能性がある(今回のアトミックリネームは単一書き込みの破損は防ぐが、複数プロセス間の「どちらが最新か」の調停までは行っていない)。現状のアーキテクチャでは問題にならないが、将来の設計変更時に思い出す必要がある。
4. **`generate_eval_testset.py`・`run_eval.py`のような一時的な別プロセスからの呼び出し**: これらのスクリプトも`get_sigmaris_jwt()`を呼ぶため、今回の修正により`.state/sigmaris_jwt_session.json`が存在していれば、常駐プロセスと同じ永続化済みトークンから正しくブートストラップできるようになる。これはPhase C-miniのbaseline取得作業自体も間接的に改善される副次効果である。
