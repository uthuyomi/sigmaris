# Phase B7 実施報告: マルチホップ質問の分解検索

対象ブランチ: `phase-b7-multihop-decomposition`(mainからfork)

## 0. baseline値の性質について(先行B群タスクと同様)

`docs/sigmaris/phase_c_mini_report.md`のbaselineおよび先行B群タスクの参考値は、
`sigmaris_decision_log`のデータ蓄積状況に依存する不確実性を含む。加えて指示書で
明示されている通り、C-miniのテストセットは現状ほぼ単発の事実確認質問中心であ
り、マルチホップ推論の効果を正しく評価できない可能性が高い。本タスクの効果確認
は、実モデルAPIでの検証ができない制約下、マルチホップ質問のテストケースによる
定性的なmock/単体テストで行った(4章)。

## 1. 分解要否判定ロジックの実装詳細

### 既存の意図分類(`classify_chat_intent`)には統合しなかった(判断根拠)

`chat_routing.py`の`classify_chat_intent()`は、**スケジュールエージェント側**
(`chat.py`、`/agent/chat/complete`等のHTTPエンドポイント経由でオーケストレータ
から呼び出される別プロセス/別関心事)の、カレンダー・移動計画・予定インポート
等のツール振り分けのための意図分類であり、記憶検索とは全く異なる関心事である。
さらに構造上、オーケストレータ(`orchestrator/service.py`)は記憶コンテキストを
**スケジュールエージェントを呼び出す前**に構築しており、`classify_chat_intent`
の判定結果はスケジュールエージェント内部でしか得られない(オーケストレータ側に
返ってくる契約がそもそも存在しない)。この分類器を流用するには、スケジュールエ
ージェントのAPI契約自体を変更してオーケストレータに意図情報を返す必要があり、
本タスクの範囲を明らかに超える過剰な変更になると判断した。

このため、**オーケストレータの記憶コンテキスト構築処理
(`_build_memory_context()`)の中に、独立した軽量なステップとして追加**した。

### 判定とサブクエリ生成を1回のLLM呼び出しに統合(判断根拠)

「分解が必要か」の判定と「サブクエリの生成」を2回に分けたLLM呼び出しにする案も
検討したが、複雑な質問のパスで往復が倍になり、レイテンシ・コストの両方で不利に
なるため、**1回のLLM呼び出しで`{"needs_decomposition": true, "sub_queries":
[...]}`または`{"needs_decomposition": false}`のいずれかを返させる設計**にした
(`multihop_search.decompose_query()`)。単純な質問では常にこの1回の呼び出しの
みで完結し、複雑な質問と判定された場合のみ追加の検索処理(下記2章)が走る。

### 重要な非対称性: 他のB群検出処理とは異なりレスポンス経路上に位置する

A3(decision)・B2(episode)・B6(topic)の検出処理は全てfire-and-forget(応答
生成後にバックグラウンドで実行)であり、ユーザーへの応答速度に影響しない。**本
タスクの分解判定はこれらと違い、応答を生成する前の記憶検索の一部として実行され
るため、そのレイテンシは直接ユーザーが体感する**。この違いを`multihop_search.
py`のモジュールdocstringに明記した(3章で詳述)。

### 専用TaskType: `QUERY_DECOMPOSITION`(新規)

B2(`EPISODE_DETECTION`)・B6(`TOPIC_DETECTION`)と同じ前例に従い、ローカルLLM
対象集合・OpenAI nanoモデル振り分け先の両方に追加した。応答経路上にあるため、こ
こでも軽量・高速なnano階層を使うことがB2/B6以上に重要になる。

## 2. サブクエリ分解・統合ロジックの実装詳細

### サブクエリ上限: 4個(`_MAX_SUBQUERIES`)

指示書の目安(2〜4個程度)に従い、上限を4とした。動機となった例(「AdFlow AI
の件で前回決めた方針」+「シグマリスの技術選定」)は2個で足りるため、4は3〜4方向
の比較質問にも対応できる余地を残しつつ、LLMが中程度に複雑な単一話題の質問を過剰
に細分化して量産することを防ぐための上限として選んだ。

### 分解の「実質1個」ガード

LLMが`needs_decomposition: true`を返しても、重複除去後にサブクエリが1個以下にな
った場合は「分解不要」(`None`)として扱う。これは分解の意味がない(単一クエリ
と同じ検索コスト)にもかかわらず追加の往復コストだけがかかる状態を避けるための
防御。

### 個別検索: 既存のB1ハイブリッド検索をそのまま再利用

各サブクエリに対して`memory_search.search_relevant_memories()`(B1のベクトル+
全文検索のハイブリッド、B17の重要度重み付け・B13の採用回数重み付けを含む)を独
立に呼び出す。**`asyncio.gather()`で並行実行**し、追加レイテンシが「サブクエリ
数×1回分の検索時間」ではなく「最も遅い1回分の検索時間」に近づくようにした(既
存のB1自体がembedding生成とtrgm検索を並行実行しているのと同じ設計思想)。

### 統合: N個のランク済みリストに対するReciprocal Rank Fusion

各サブクエリの検索結果は、それ自体が既にB1/B17/B13の重み付けで並べ替え済みのラ
ンク付きリストである。これらN個のリストを1つに統合する際、単純な連結+重複排除
(先勝ち)ではなく、**B1の`_merge_hybrid_results()`と同じReciprocal Rank Fusion
の考え方を、2リストではなくNリストに一般化した`_rrf_merge_ranked_lists()`**を実
装した(判断根拠: 単純結合は「たまたま先に処理されたサブクエリの結果」に偏る
が、RRFは各リスト内の順位を公平に合算するため、複数のサブクエリで共通して上位に
現れた記憶が自然に優先される)。定数`_RRF_K`はB1が既に定義しているものを
`memory_search`からそのままimportし、同じ意味の定数を重複定義しないようにした。

### 統合結果の件数上限: 8件(`_MULTIHOP_RESULT_LIMIT`)

単一クエリのデフォルト`limit=5`より広めに設定した(判断根拠: マルチホップ質問
は単一話題の質問より本質的に多くの異なる事実をカバーする必要があるため)が、
「サブクエリ数×5」(最大20件)のような無制限の膨張は防ぎ、注入されるコンテキス
トのサイズをある程度予測可能な範囲に収めた。

### Phase B6の話題コンテキストの活用方法(判断根拠)

指示書で「B6の話題遷移トラッキングが有用な場合は活用してよい」とされていたた
め、`decompose_query()`のプロンプトに「直近の話題の推移」という参考情報を追加で
きるようにした。**ただし新たに`topic_tracker.get_recent_topics()`を追加で呼び
出すことはせず、`orchestrator/service.py`が既存の`asyncio.gather`で取得済みの
`current_topic`/`previous_topic`(最大2件)をそのまま再利用する**設計にした
(`_topic_labels_for_hint()`)。理由: この情報はあくまで「無理に使う必要はない」
参考情報であり、追加のDB往復コストをかけてまでB6を深く遡る必要性は薄いと判断し
た。B6自体の設計変更(指示書で範囲外と明示)は一切行っていない。

## 3. レイテンシへの影響

実モデルAPIでの検証ができないため、実測値ではなく設計上の見積もりを記載する。

### 単純な質問のパス(要件2)

変更前: `search_relevant_memories()`を1回呼ぶのみ。
変更後: `decompose_query()`(nano階層のLLM呼び出し1回、`needs_decomposition:
false`判定)→ 分解不要と判明 → 従来通り`search_relevant_memories()`を1回呼ぶ。

**追加コストはnano階層のLLM呼び出し1回分のみ**(A3/B2/B6の各検出処理が既に使っ
ている同じ軽量モデル階層と同等のレイテンシ、おおよそ数百ms程度と見積もる)。こ
れは応答経路上に位置するため(2章で述べた通り)ユーザーが体感する遅延が増える
唯一のB群機能であり、この点は要件2の「最小限に抑える」という要求に対して、ゼロ
ではなく「1回分の軽量LLM呼び出し」が現実的な下限であると整理した。

### 複雑な質問のパス

`decompose_query()`のLLM呼び出し1回(単純な質問と同じ、判定とサブクエリ生成を
1回にまとめているため往復は増えない)+ 最大4個のサブクエリ検索(`asyncio.
gather`で並行実行、追加レイテンシは検索1回分にほぼ収束)+ 統合処理(全てメモリ
上の計算、無視できるコスト)。

複雑な質問のパス全体としては、既存の単一検索パスと比べて「LLM呼び出し1回分」
+「検索1回分弱の並行実行オーバーヘッド」が増える見積もりであり、サブクエリ数に
比例して線形に遅くなることはない設計にした。

## 4. テスト結果

実モデルAPIでの検証はできないため、mock/単体テストで確認した(3ファイル、計22
ケース、B2/B3/B6と同じくスクラッチテストとして作成・実行、backend/tests配下に
はコミットしていない)。

### `decompose_query()`(8ケース)
- 単純な質問で`None`(分解不要)が返ること
- 複雑な質問でサブクエリのリストが正しく返ること
- サブクエリが上限4個に切り詰められること
- 重複するサブクエリが除去されること
- 重複除去の結果1個以下になった場合は分解不要として扱われること
- 空の質問ではLLMが一切呼ばれないこと
- LLM呼び出し失敗時は例外を投げず分解不要として扱われること
- 話題ヒントがプロンプトに正しく含まれること

### `_rrf_merge_ranked_lists()`(3ケース)
- 複数のサブクエリ結果に共通して現れる記憶が、1つのリストにしか現れない記憶よ
  り上位にランクされること
- 件数上限が正しく機能すること
- idを持たない行がスキップされること

### `search_with_decomposition()`(5ケース)
- 単純な質問では単一検索パスがそのまま使われること(既存動作と完全互換)
- 複雑な質問で複数のサブクエリ検索が並行実行され、結果が統合されること
- 1つのサブクエリ検索が失敗しても、他のサブクエリの結果は失われないこと
- 全サブクエリ検索が失敗した場合、空リストを返すが「分解は行われた」ことは記録
  されること
- 統合結果が`_MULTIHOP_RESULT_LIMIT`(8件)以内に収まること

### `orchestrator/service.py`の配線(6ケース)
- `_build_memory_context()`が(直接`search_relevant_memories`ではなく)`search_
  with_decomposition()`を呼ぶこと、話題ラベルのヒントが正しく渡されること
- 検索失敗時も例外が伝播しないこと(既存の`try/except`の維持確認)
- `_topic_labels_for_hint()`が現在・直前の話題を時系列順(古い→新しい)で正しく
  リスト化すること、空白ラベルを除外すること

### 既存機能への非破壊確認(要件4)
`backend/tests/`の既存8テストに加え、B2/B3/B6のスクラッチテスト・facts キャッシ
ュ修正のテストを含めた全76ケースを再実行し、全て成功することを確認した。

```
22 passed (B7新規テスト)
54 passed (B2/B3/B6スクラッチテスト・facts cache修正テスト再実行)
8 passed (既存回帰テスト、backend/tests/)
```

## 5. 気づいた懸念点・次のB機能(B8: 時間認識型リランキング)に影響しそうな発見

- **応答経路上のレイテンシ追加は、この機能特有の性質であり、次にレイテンシ影響
  のある機能を追加する場合は同様の整理(fire-and-forgetかどうかの明示)が必要**:
  A3/B2/B3/B6は全てfire-and-forgetで応答速度に影響しないが、本タスクは初めて
  「応答速度に直接影響する」B群機能になった。B8(時間認識型リランキング)がもし
  検索結果のリランキングに追加のLLM呼び出しを要する設計になる場合、同じ非対称性
  が生じる可能性が高く、レイテンシ予算の合計(既にB7で1回分のLLM呼び出しが応答
  経路に乗っている)を意識する必要がある。
- **サブクエリの分解粒度はLLM任せであり、意味的に重複するがテキストとしては異な
  るサブクエリ(例:「AdFlow AIの方針」と「AdFlow AIについて決めたこと」)は現状
  の完全一致による重複排除をすり抜ける**: 実運用でこれが問題になる場合、B6で見
  送った軽量な類似度チェックの追加を、こちらの重複排除ロジックにも検討する余地
  がある(ただし過剰実装を避けるという指示書の方針に従い、今回は実装していな
  い)。
- **`_MULTIHOP_RESULT_LIMIT=8`は経験的な調整をしていない初期値**: B13/B17で確立
  した「rank-gap flip-testによる重みの実測」という手法は、今回は検索結果の「件
  数」の調整であり同じ手法は直接使えない。実運用でコンテキストが長すぎる/短すぎ
  るといった問題が観測された場合、この値の調整を検討されたい。
- **`decompose_query()`の判定基準(「複数の異なる記憶を組み合わせる必要がある
  か」)は、B8が仮に「時間軸を跨いだ質問」(例:「先月と今月で状況はどう変わっ
  た?」)を扱う場合、時間的な分解(過去のある時点 vs 現在)もこの分解ロジックの
  対象に含めるべきか、B8側で別途扱うべきかの整理が必要になる可能性がある。今回
  は「異なる話題」の分解のみを対象とし、「同じ話題の異なる時点」の分解は明示的
  にスコープ外とした。**
