# Phase B 完了サマリー(B1〜B17、全17機能)

本ドキュメントは、Phase B(記憶・RAG改善ロードマップ、全17機能)の完了をもっ
て作成する最終まとめである。各機能の実装詳細・判断根拠・テスト結果は個別の
`docs/sigmaris/phase_bN_report.md`を参照のこと。本サマリーは技術的詳細を繰り
返さず、全体像・技術的教訓・残存する懸念事項・Phase C-full着手前の対応事項に
焦点を当てる。

## 2. 実行順序と、ロードマップとの整合性について(重要な経緯)

本セッションで実際に実行された順序は次の通り:

```
B1 → B4 → B17 → B14 → B13 → B2/B3/B6 → B7/B8 → B10/B11/B12 → B15/B16 → B9 → B5
```

B9完了報告(`phase_b9_report.md`)の時点で、**B5がこの実行順序に一度も含まれ
ていなかったことが判明した**。B5はPhase B群で唯一のフロントエンドタスク(開発
者向けダッシュボード)であり、他の16機能がすべてバックエンドの記憶ロジックで
あったのとは性質が異なる。今回B5を実施したことで、B1〜B17の全17機能が完了し
た。今後同種の抜け漏れを防ぐため、フェーズ完了報告のたびにロードマップとの突
き合わせを行うことを推奨する(`phase_b5_report.md`5章にも記載)。

## 3. 技術的な教訓(全体を通じて)

1. **応答経路上のレイテンシは「予算」として累積管理する必要がある**: B7・B10
   の2つのみが新規LLM呼び出しを追加し、それ以降の全機能(B11〜B16、B9)がこの
   制約を厳格に守った。この「一度導入したレイテンシ予算を増やさない」という規
   律は、後続タスクへの申し送りが正しく機能した好例だった。
2. **複数の重み付けを合成する機能は、必ず実測(パターン: 隣接ランク間の最大逆
   転ギャップ)で境界を検証する**(B13・B8・B15で確立): RRFスコアは差が小さい
   ため、小さな重みでも合成すると想定以上にランキングを動かしうる。
3. **fire-and-forgetの「保留状態→後続ターンでの消費」パターンは高い再利用性
   を持つ**: B3で確立され、B15(スレッド単位)・B16(グローバル)で異なるスコ
   ープに適用された。
4. **`importance_score`(B17)はカテゴリ固定値の代理指標であり、個別学習値では
   ない**という性質は、B12・B16・B9・B5の複数タスクに影響した。「重要度」を扱
   う機能を実装する際は、その数値が本当に個別学習された値なのか、カテゴリ固定
   値の代理なのかを都度確認する必要がある。
5. **フロントエンドタスク(B5)は、バックエンドタスクと検証の質が根本的に異な
   る**: mock/単体テストで論理的な正しさは検証できても、実際の画面表示・操作
   感はブラウザでの確認が必須であり、型検査・lintだけでは不十分。今後Phase C
   でフロントエンドタスクが増える場合、B群の標準的な検証プロセス(mock単体テ
   ストで十分)をそのまま適用できない。
6. **指示書が前提とする永続化状態が、実際には存在しない(プロセス内メモリのみ
   等)ケースがある**(B5で発覚): B3の確認状態はDBに永続化されておらずプロセ
   ス内の辞書のみで管理されていたが、この制約はB5の指示書作成時点で反映されて
   いなかった。複数フェーズにまたがる前提を含むタスクでは、対象データの永続化
   状況を都度確認する必要がある。
7. **`similarity`フィールドは、ランキング経路によらず一貫してスケールされた
   唯一の信号である**(B11の発見): 最終スコアの意味はtier/pathによって変わる
   が、`similarity`自体はどのランキング加重・リランクによっても上書きされな
   い。
8. **マイグレーションの「DROP FUNCTION + CREATE FUNCTION」が必要になるケース
   の繰り返し発生**: `RETURNS TABLE`の列を追加する変更のたびに4回発生した
   (B1→B17→B13→B8)。

## 4. 残っている懸念事項(総まとめ)

1. **5〜6個の類似パイプラインの重複**(`phase_b16_report.md`で最初に指摘、
   `phase_b9_report.md`で6個目として再確認): B2・B6・B14・B15・B16・B9が、そ
   れぞれ独立に「週次抽出→グローバルservice-roleテーブル→TTLキャッシュ注
   入」という同型のパイプラインを実装している。共通化するリファクタリングは
   B群の範囲外として先送りされ続けており、**Phase C-full着手前、または独立し
   たリファクタリングタスクとして改めて検討することを強く推奨する**。
2. **実モデルAPIでの効果測定が一度も行われていない**: `OPENAI_API_KEY`・SSH/
   サーバーアクセスのいずれも本セッションを通じて一切得られなかったため、B1〜
   B17の全機能がmock/単体テストによる論理的な検証のみで実装されている。
3. **未適用マイグレーションの累積**(13件、B1〜B9で作成分。B5はマイグレーショ
   ン不要のため含まれない): `202607060028`〜`202607180040`。適用順序に依存関
   係があるため、作成順に適用することを推奨する。
4. **`sigmaris_decision_log`等の実際のデータ蓄積量が未確認**: B13〜B16の複数
   機能が依存する最低証拠件数のゲーティングが、実運用でどの程度「データ不足」
   に倒れているか未確認。
5. **B5ダッシュボードの実ブラウザでの表示確認が未実施**: 型検査・lintは通過し
   ているが、実データでの表示・フィルタ/ソート動作の実機確認は運用者側で行う
   必要がある。
6. **`/chat`表示崩れの調査(本セッション序盤、B群とは別系統)が未解決のまま残
   っている**: 追加指示がないまま保留。

## 5. Phase C-full着手前に対応しておくべき事項

1. **13件のマイグレーションを全て、作成順に適用すること**(最優先): これが行
   われない限り、B群の実装コードの大部分は本番環境で実際には動作しない。
2. **`run_eval.py`を実マイグレーション適用後・実モデルAPI環境で実行し、
   C-miniの3指標の実測値を取得すること**。
3. **5〜6パイプラインの重複を解消するリファクタリングを検討すること**: 共通化
   の候補は「週次抽出ジョブの雛形」「service-role専用テーブルへのget-or-
   create/upsertヘルパー」「TTLキャッシュ付き注入ヘルパー」の3点。
4. **応答経路上のLLM呼び出し数(B7・B10の2つ)の実測レイテンシを確認するこ
   と**。
5. **B5ダッシュボードを実データで一度ブラウザ確認すること**。
6. **Phase C以降の各タスクでも、ロードマップとの突き合わせを都度行うこと**
   (B5のスコープ抜けの再発防止)。
