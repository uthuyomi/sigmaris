対象ブランチ: `phase-b12-post-retrieval-compression`(mainからfork)

`docs/sigmaris/phase_c_mini_report.md`のbaselineおよび先行B群タスクの参考値は不

## 1. 圧縮方式の実装詳細: ルールベースを採用(LLMベースは不採用)

### 判断根拠

指示書の指示通り、まずルールベースの手法を検討し、それで要件を満たせると判断し
たため、**LLMベースの要約は採用しなかった**。理由:

- B7・B10は既に応答経路上に新規LLM呼び出しを追加しており、B11はこれ以上増やさ
  ないという制約を新規呼び出しゼロで達成した。本タスクで要約用のLLM呼び出しを追
  加すると、B7・B10のような「判定のための1回」ではなく、**トークン閾値を超える
  ターン全てで実行される要約呼び出し**になり、B7・B10より悪いレイテンシ特性にな
  る(判定は常に軽量だが、要約はテキスト量に応じて重くなりうる)。
- 文字数上限・キーワード近傍抽出というルールベースの手法だけで、要件1〜3(重要
  度に応じた圧縮強度・情報欠落の回避)を十分満たせると判断した。

### トークン推定: 新規ライブラリ(tiktoken等)は追加していない

`docs/sigmaris/phase_a2_report.md`がトークン数比較にtiktokenを使っていたため確認
したところ、**`tiktoken`は`backend/pyproject.toml`の依存関係に含まれておらず、
`.venv`にも実際にインストールされていない**ことを確認した(Phase A2の比較は一
時的なローカル分析だったと考えられる)。B10で確立した「新規依存関係の追加は慎重
に検討する」という方針を踏まえ、本タスクでも新規ライブラリは追加せず、**文字数
ベースの近似**(日本語主体の混在テキストに対して1トークン≒2文字という単純な比
率)でトークン量を見積もる方式にした。この近似は正確ではないが、圧縮の要否判定
(閾値超過の有無)という用途には十分と判断した。正確な値が必要な場合は、Phase
A2と同じ手法(tiktokenでの一時的な比較)を運用者側で実施できる。

### 圧縮方法: 文の重要度スコアリング(B1のtrigramと同じ考え方をPython側で踏襲)

記憶の`value`が予算(後述)を超える場合、句点等で文単位に分割し、各文と質問文の
**文字2-gram(bigram)の重複度**でスコアリングし、質問との関連性が高い文を優先的
に残す。これはB1が`pg_trgm`(文字3-gram)で採用した「日本語には単語境界がないた
め文字n-gramで類似度を測る」という考え方をそのままPython側の文選択に応用したも
のである(bigramにした理由: B1が扱う「事実全体同士の検索」よりも、本タスクが扱
う「1つの事実内の短い文同士の比較」の方が対象テキストが短く、3-gramではスコアが
疎になりすぎると判断したため)。関連する文が見つからない(文構造がない、または
重複が皆無)場合は、単純な先頭からの文字数切り詰めにフォールバックする。

## 2. 重要度との統合方法(B17との連携詳細)

B17の申し送り通り、`importance_score`(B17で判明した通り、DBトリガーがカテゴリ
単位で設定する粗い代理指標)を圧縮対象の選定にそのまま使った。B17が確認したカテ
ゴリ別の値(`goals=1.0, health=0.9, profile=0.8, relationships=0.8, finance=
work=personality=timeline=0.7, lifestyle=0.6, preferences=0.5, devices=
environment=0.4`)を踏まえ、以下の3段階を設定した:

| importance_score | 該当カテゴリ | 扱い |
|---|---|---|
| 0.8以上 | goals・health・profile・relationships | **圧縮対象外**(要件1) |
| 0.5以上0.8未満 | finance・work・personality・timeline・lifestyle・preferences | 中程度の予算(150文字) |
| 0.5未満 | devices・environment | 積極的な圧縮(60文字、要件2) |

`importance_score`が欠落している行は、B17・B13と同じ既存の防御パターン
(`_DEFAULT_IMPORTANCE_SCORE=0.5`、`memory_search.py`から再利用)で中間扱いにフ
ォールバックする。

## 3. 適用範囲の制御ロジック(閾値とその根拠)

**トークン閾値: 400**(推定)。この値を上回った場合のみ圧縮ロジック全体が発動す
る(要件3、「常時圧縮すると通常時の応答品質が不要に劣化するリスク」を避けるた
め)。

根拠: 通常の単一クエリ検索(B1、5件)で、各事実の`value`が数語程度の短い事実
(例:「登山」「東京都在住」)である典型的なケースでは、5件分を合計しても400ト
ークンには遠く及ばない(実測で1件あたり十数トークン程度)。一方、B7のマルチホッ
プ経路(最大8件)や、値が長い自由記述のノート的な事実が混ざった場合には、400ト
ークンを超えるケースが生じうる。つまりこの閾値は「通常の単発質問では発動せず、
マルチホップや長文ノートが絡む場合にのみ発動する」という、指示書が求める段階的
適用そのものを実現している。

## 4. トークン削減効果の実測

トークン推定関数自体はPythonの純粋な計算であり(LLM API呼び出しを伴わない)、
以下は**実際に本タスクの実装コードを実行して得られた実測値**である(見積もりで
はない)。

代表的なマルチホップ相当のシナリオ(8件、重要度混在: goals/health各1件(圧縮対
象外)・finance/lifestyle/preferences各1件(中程度)・devices2件+environment1件
(積極的)、各事実の`value`は約248文字の自由記述ノート)で実測:

```
was_compressed: True
圧縮前: 2399文字(推定約1200トークン)
圧縮後: 1547文字(推定約774トークン)
削減率: 35.5%
```

goals・healthの2件は圧縮対象外のため全文がそのまま残っており(要件1)、情報の
欠落は生じていない。残り6件のうち中程度・積極的圧縮の対象になった事実は、質問
文(「富士山の予定について」)との関連が高い文が優先的に残された。

## 5. レイテンシへの影響(最重要要件)

`compress_memories_if_needed()`は純粋なPython関数であり、LLM呼び出し・DB呼び出
し・HTTP呼び出しを一切含まない。B11と同じ検証手法(`get_llm_router`を直接mock
し、`_build_memory_context()`実行後に一切呼ばれていないことを確認するテスト)
を用いて、本タスクのコードがLLMルーターに一切触れていないことを実装レベルで検
証した(6章)。**応答経路上のレイテンシは変化しない**。

## 6. テスト結果

実モデルAPIでの検証はできないため、mock/単体テストで確認した(2ファイル、計15
ケース、これまでのB群と同じくスクラッチテストとして作成・実行、backend/tests
配下にはコミットしていない)。

### `estimate_tokens()`・`compress_memories_if_needed()`(9ケース)
- 空文字列は0トークンと推定されること
- 長いテキストほど推定トークン数が増えること
- 合計トークン量が閾値以下の場合、圧縮処理自体が発動せず、入力がそのまま(同一
  オブジェクトとして)返されること
- 閾値を超える場合でも、全て高重要度カテゴリであれば一切圧縮されないこと(要件
  1)
- 低重要度の事実が積極的に(60文字予算まで)圧縮されること(要件2)
- 中重要度の事実が中程度(150文字予算)まで圧縮されること
- 圧縮時、質問文と関連性の高い文が優先的に残されること(スニペット抽出の直接確
  認)
- 文構造がないテキストでは、単純な先頭切り詰め(末尾に「…」)にフォールバック
  すること
- 圧縮処理が元の辞書オブジェクトを一切変更しないこと(コピーを返す設計の確認)

### `_build_memory_context()`の配線(6ケース)
- 短い検索結果はそのまま(無変更で)コンテキストに反映されること(過剰圧縮の防
  止確認)
- 大きく低重要度な結果セットが、実際に最終コンテキスト文字列でも圧縮されている
  こと(圧縮前の想定文字数より明確に短くなることを直接確認)
- 高重要度の事実は、他の事実が圧縮されていても全文がそのままコンテキストに残る
  こと(要件3、情報欠落防止の統合確認)
- 検索失敗時も例外を投げないこと
- **`get_llm_router`が一切呼ばれないこと(要件4の直接検証)**

### 既存機能への非破壊確認(要件6)
`backend/tests/`の既存8テストを含む、これまでの全B群スクラッチテストを合わせた
143ケースを再実行し、全て成功することを確認した。B11の20ケースも無変更のまま全
て成功しており、B11の確信度分類ロジック(圧縮前の`relevant`に対して実行される順
序を維持)への影響がないことを確認した。

```
15 passed (B12新規テスト)
128 passed (B2/B3/B6/B7/B8/B10/B11スクラッチテスト・facts cache修正テスト・
            既存回帰テスト再実行)
```

## 7. 気づいた懸念点・次のB機能(B15: 個人別放棄閾値)に影響しそうな発見

- **文字数ベースのトークン推定は近似であり、実際のトークナイザーとの乖離が未検
  証**: 日本語・英語混在のテキスト(製品名等の固有名詞を含む事実)では、1トーク
  ン≒2文字という比率が実際とどれだけ乖離するかは、tiktoken等での実測(運用者側)
  が必要。この乖離が大きい場合、閾値400の妥当性も再検討が必要になる可能性があ
  る。
- **B11(校正された放棄判定)との実行順序を明示的に固定した**: 確信度の判定は圧
  縮前の`relevant`に対して行い、圧縮は表示直前の最後の変換ステップとして適用す
  る設計にした。これにより、B11の判定ロジックが将来変更されても、圧縮の有無に左
  右されない独立性を保っている。B15(個人別放棄閾値)がB11の閾値を人物ごとにカ
  スタマイズする設計になる場合も、この実行順序(B11の判定 → B12の圧縮 → 描画)
  は変更する必要がないはずである。
- **圧縮のスニペット抽出は文単位が最小粒度であり、1文の途中で意味的に重要な部分
  と不要な部分が混在している場合には対応できない**: 現状のルールベース手法の限
  界であり、より精密な圧縮が必要になった場合はLLMベースの要約を再検討する余地が
  あるが、その場合は本タスクで避けた応答経路上のレイテンシ増加という代償を払う
  ことになる点に注意が必要。
- **`_COMPRESSION_TRIGGER_TOKENS=400`は、B7のマルチホップ経路(最大8件)を想定
  した値であり、実際の運用でこの経路がどの程度の頻度で使われるか次第で、圧縮が
  発動する頻度も変わる**: B7の報告書でも「候補数がlimitを超える場合のみ発火す
  る」処理の実際の発火頻度が未計測だったのと同様の懸念であり、運用データでの実
  測を推奨する。
