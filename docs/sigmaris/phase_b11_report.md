## 1. 閾値設定の実装詳細

### 既存スコア分布の確認: 「最終ランキングスコア」ではなく`similarity`を採用した理由

まず、B1〜B10を経た「最終的なランキングスコア」がどのような分布を持つかを調査
した。結果、**単一の一貫したスケールを持つ「確信度」の数値はどこにも存在しない
**ことが判明した:

- `_merge_hybrid_results()`の「近接一致(trgm高類似度)」ティアは生の類似度
  (0.15〜1.0程度)をベーススコアとして使う。
- 「通常」ティアはRRF融合スコア(`1/(_RRF_K+rank)`の合算、実測でおよそ0.008〜
  0.02程度の極めて小さい値)をベーススコアとして使う。
- B10のリランキングは、LLMの判定順位を**新しいRRFスコアに変換し直す**ため、元の
  ベーススコアはリランキング後には使われなくなる。

つまり「最終的にソートに使われた値」はティア・パスによって意味もスケールも異な
り、固定閾値を意味のある形で適用できる対象ではない。

一方、各検索結果の行自体が持つ**`similarity`列**(ベクトルのコサイン類似度、ま
たはtrgm類似度)は、B17・B13・B8の重み付けやB10のリランキングのいずれでも**一切
上書きされない**(これらは全て`base_score`から派生した乗数でソート用のスコアを
計算するだけで、行データ自体は変更しない)ことを実装コードの確認により突き止め
た。この`similarity`は常に[0, 1]の範囲を持ち、どのティア・パスを経ても意味が一
定であるため、これを閾値判定の対象とした。

### 具体的な閾値

`search_relevant_memories()`は、本プロジェクト全体を通じて常に`threshold=0.7`
(ベクトル検索の足切り)で呼ばれている(`orchestrator/service.py`・
`multihop_search.py`のいずれも)。この既存の足切り値を基準に、そこからのマージ
ンとして閾値を設定した:

- **ベクトル・通常質問: 0.78**(足切りから+0.08のマージン)。「一応検索の足切
  りは通過したが、かろうじて」というゾーンを捉える、控えめな値。
- **ベクトル・厳格(マルチホップまたは時系列質問): 0.85**(+0.15のマージン)。
  マルチホップ質問は複数の事実を組み合わせて回答を合成するため、片方の裏付けが
  弱いと合成された回答全体のリスクが増す。時系列質問はB8が対処してきた「今も有
  効か」という誤りやすい領域そのものであり、断定する前により高い確信度を要求す
  るのが妥当と判断した。
- **trgm: 0.5**(B1が既に定義している`_TRGM_HIGH_CONFIDENCE_SIMILARITY`をそのま
  ま再利用)。trgm類似度はベクトルのコサイン類似度のような連続的な「足切りから
  どれだけ離れているか」という勾配を持たないため、新たな閾値を発明せず、B1が既
  に「近接一致」の境界として検証済みの値をそのまま流用した。マルチホップ・時系
  列質問でもtrgmの閾値は据え置いた(これ以上厳しくすると正当な完全一致キーワー
  ドヒットまで除外してしまうリスクの方が大きいと判断したため)。

### 質問カテゴリの再利用(新規判定なし)

「質問のカテゴリ」は、B7の`decompose_query()`が既に判定している
`needs_decomposition`(マルチホップかどうか)・`time_sensitive`(時系列質問かど
うか)をそのまま再利用した。これらは本タスク以前から毎ターン計算済みであり、本
タスクのために新たに判定する必要は一切ない。

### 固定閾値のみ、自動学習は実装していない

指示書の指示通り、閾値は固定値のみとし、過去の正誤ログからの動的調整は実装して
いない。将来の拡張余地として、`memory_confidence.py`の各閾値は名前付き定数とし
て独立させてあり、動的な調整ロジックを追加する場合もこれらの定数を差し替えるだ
けで済む設計にしてある。

## 2. レイテンシに影響を与えない実装方法の詳細(最重要要件)

`memory_confidence.py`の`classify_confidence_tier()`・`confidence_guidance_
note()`はいずれも**純粋なPython関数**であり、LLM呼び出し・DB呼び出し・HTTP呼び
出しを一切含まない。入力は`search_with_decomposition()`が(本タスク以前から)既
に返している`memories`リストと、同じくB7が既に計算している`was_decomposed`・
`time_sensitive`のみである。

これを担保するため、`test_b11_orchestrator_wiring.py`に
`NoNewLlmCallTests`という専用のテストクラスを設け、`get_llm_router`を直接mockし
た上で`_build_memory_context()`を呼び出し、`search_with_decomposition()`自体は
モックで置き換えているにもかかわらず`get_llm_router`が一切呼ばれないことを明示
的に確認した。これにより「本タスクのコード自体がLLMルーターに触れていない」こ
とを実装レベルで検証している。

`search_with_decomposition()`の戻り値を`(memories, was_decomposed)`の2要素タプ
ルから`(memories, was_decomposed, time_sensitive)`の3要素タプルに変更したが、
これは**戻り値の型を拡張しただけ**であり、`decompose_query()`が実行するLLM呼び
出しの回数・内容は一切変更していない(既に計算済みの`analysis.time_sensitive`
を追加で返すようにしただけ)。

## 3. 応答への反映方法

`orchestrator/service.py`の`_build_memory_context()`に、`search_with_
decomposition()`の呼び出し直後、以下のロジックを追加した:

```python
tier = classify_confidence_tier(
    relevant, is_multihop=was_decomposed, is_time_sensitive=time_sensitive
)
guidance = confidence_guidance_note(tier)
if guidance:
    relevant_context = f"{relevant_context}\n{guidance}" if relevant_context else guidance
```

`confidence_guidance_note()`は3種類のノートを返す:

- **confident**: `None`(注意文を一切追加しない — 既存動作から完全に無変更。過
  剰にヘッジしないという要件を、コード上「何も足さない」という形で直接担保して
  いる)。
- **low_confidence**: persona.md 5章の「仮説層」の言い回し(「もしかしたら〜か
  もしれません」「まだ確証はないんですが」)を指示する注意文。B14で確立した
  persona.md 5章へのマッピングパターンを踏襲した。
- **no_evidence**: 「その件については、まだ十分な情報がありません」という正直
  な表明を指示する注意文。

### 「検索結果0件だが単なる雑談だった」場合の扱い(判断根拠)

`_build_memory_context()`は現状、ユーザーの最新発話が空でない限り常に検索を実行
する。これは単なる雑談(「今日はいい天気ですね」等)でも検索が走り、結果が0件に
なりうることを意味する。このとき無条件に「情報がありません」という注意文を注入
すると、雑談の応答に不自然な影響を与えるリスクがある。

この曖昧さを解決するために、新たな判定ロジック(ヒューリスティックや追加のLLM呼
び出し)を導入することも検討したが、**追加のLLM呼び出しは本タスクの最優先制約
に反する**ため採用しなかった。代わりに、`no_evidence`ノート自体の文面に「単なる
雑談であれば、この注意は無視して構いません」という一文を明示的に含めることで、
この判断を**どのみち今回のターンを処理する既存の応答生成LLM自身に委ねる**設計に
した。これにより新たなAPI呼び出しは一切発生せず(既に送信されるプロンプトに数
十トークン追加されるだけ)、かつ雑談への悪影響リスクを実質的に回避できると判断
した。

## 4. テスト結果

実モデルAPIでの検証はできないため、mock/単体テストで確認した(2ファイル、計20
ケース、これまでのB群と同じくスクラッチテストとして作成・実行、backend/tests
配下にはコミットしていない)。

### `classify_confidence_tier()`・`confidence_guidance_note()`(14ケース)
- 検索結果が0件の場合`no_evidence`になること
- ベクトル類似度が通常閾値(0.78)を上回れば`confident`、下回れば
  `low_confidence`になること
- 通常閾値と厳格閾値(0.85)の中間の類似度が、単純な質問では`confident`、マルチ
  ホップ質問・時系列質問では`low_confidence`に切り替わること(カテゴリ別閾値の
  直接確認)
- `match_source="both"`がベクトルスケールで扱われること
- trgm類似度がB1の既存閾値(0.5)を境に正しく分類されること(カテゴリによらず
  同一閾値であることも確認)
- 複数の結果がある場合、先頭(最上位)の結果のみが判定に使われること(2位以下
  に強い結果があっても、最上位が弱ければhedgeされること)
- `similarity`欠落時は0扱いとなり安全側(hedge)に倒れること
- 各tierに対応する注意文の内容確認(仮説層の言い回し、正直な表明、「雑談なら無
  視してよい」の明記)

### `_build_memory_context()`の配線(6ケース)
- 十分な確信度の結果には注意文が一切追加されないこと(過剰ヘッジ防止の確認)
- 弱い結果に仮説層の注意文が追加されること
- 検索結果0件で正直な注意文が追加されること
- マルチホップと判定された質問で、単純質問なら`confident`になる類似度でも
  hedgeされること(厳格閾値が実際に効くことの確認)
- 検索失敗時も例外を投げず、確信度判定自体がスキップされること
- **`get_llm_router`が一切呼ばれないこと(最重要要件3の直接検証)**

### 既存のB7スクラッチテストの戻り値変更への追従
`search_with_decomposition()`の戻り値が3要素タプルに変わったことに伴い、
`test_b7_search_with_decomposition.py`・`test_b7_orchestrator_wiring.py`を更新
し、既存の16ケース全てが新しい戻り値の形でも引き続き成功することを確認した。

### 既存機能への非破壊確認(要件4)
`backend/tests/`の既存8テストを含む、これまでの全B群スクラッチテストを合わせた
128ケースを再実行し、全て成功することを確認した。

```
20 passed (B11新規テスト)
108 passed (B2/B3/B6/B7/B8/B10スクラッチテスト・facts cache修正テスト・
            既存回帰テスト再実行、戻り値変更対応含む)
```

## 5. 気づいた懸念点・次のB機能(B12: 検索後の圧縮)に影響しそうな発見

- **「最終ランキングスコアには一貫した意味がない」という発見は、B12(検索後の
  圧縮)にも影響しうる**: もしB12が「圧縮時にどの記憶を優先的に残すか」を最終
  スコアに基づいて判断する設計になる場合、本タスクと同じ理由で、最終スコアでは
  なく`similarity`(または本タスクで確立した`confidence_tier`の概念)を判断軸に
  する方が一貫性があるかもしれない。
- **`no_evidence`ノートの「雑談なら無視してよい」という設計は、応答生成LLM自身
  の判断力に依存している**: 実モデルでの検証ができないため、実際にこの文言が意
  図通り機能するか(雑談で不自然にならず、かつ本当に知らないことを聞かれたとき
  は正直に答えるか)は、運用者側での実地確認が必要。もし雑談時に不自然な言及が
  頻発するようであれば、B11の追加のLLM呼び出し禁止という制約を緩め、既存のB7の
  分解要否判定に「そもそも記憶に基づく回答が必要な質問か」という軽量なフラグを
  追加する、という統合的な対応(B7の呼び出しに1フィールド追加するだけなので新
  規呼び出しにはならない)を将来検討する余地がある。
- **応答経路上のLLM呼び出し数は、B10までで2回(B7・B10)のまま据え置かれた**:
  本タスクは指示書の最優先制約を完全に守り、新規呼び出しを一切追加しなかった。
  B12以降でも、可能な限りこの「応答経路上の呼び出し数を増やさない」という制約
  を意識した設計が望ましい。
- **閾値(0.78/0.85/0.5)は実測データではなく、既存の足切り値からのマージンとい
  う論理的な根拠に基づく初期値**: 実際の検索結果の類似度分布が想定と異なる場合
  (例えば大半の正解が0.9以上に集中している、あるいは0.7台前半にも正解が多い等)、
  運用データでの再調整が必要になる可能性がある。
